Production Ops

The day-2 job: keeping a fleet of non-deterministic agents observable, affordable, and reliable. Topic 7 taught you to grade an agent offline; this is how you watch it in production, at scale, without going broke or blind.
~11 min read · 1 exercise at the end
The one idea on this page You can’t debug what you can’t trace. A status code tells you the endpoint answered; it says nothing about why the agent did what it did. Production ops for agents rests on one substrate — a span for every decision, tool call, and model call — and everything else (online evals, cost attribution, alerting) is a query over those traces. And you alert on deltas, not absolutes, because a non-deterministic system has no fixed “correct” number.

1.Tracing: a span per decision

The accepted floor in 2026 is a span for every step, not a health check on the endpoint. One session fans out to dozens of model calls and tool calls; you need to see the whole chain:

── one session, one trace ─────────────────────────────────
invoke_agent "why is checkout slow?" 1,240 tok $0.004
  chat plan next step 890 tok $0.003
  execute_tool grep(logs) —
  chat interpret matches 2,100 tok $0.007
  execute_tool mcp: db.explain(query) ← nests via trace context
  chat conclude 640 tok $0.002
trace-root tags: user=42 · tenant=acme · task=perf-debug
a

OpenTelemetry GenAI is the standard — and still experimental

Instrument to the OTel GenAI semantic conventions (invoke_agent, chat, execute_tool spans; gen_ai.usage.*_tokens metrics), not a vendor SDK, so backends stay swappable. Caveat: as of mid-2026 the GenAI and MCP conventions are still “Development” — pin the opt-in flag and expect attribute churn.

b

One trace across many agents

Propagate trace context across every agent-to-agent and agent-to-tool hop so a subagent’s spans nest under the orchestrator. This is what makes a 15×-fanout multi-agent run debuggable instead of a pile of orphaned spans. MCP servers enrich the same trace via W3C Trace Context.

c

Tail sampling, never head

Failures are rare and high-value — head sampling drops them at emit time. Sample at the tail: keep 100% of errors, 100% of low eval-score traces, 100% of high-cost traces, and 1–10% of clean ones for baselines. This also keeps observability cost sublinear.

2.Evals in production: the online layer

Topic 7 was the offline test suite. In production you run the same judgments on live traffic — monitoring, not testing:

1

Deterministic on 100%, judge on a sample

Cheap rule checks (schema, business rules, guardrail pass/fail) run on all traffic. The LLM judge runs async on ~5% of sessions — never synchronously on the critical path, or you double latency and cost. Treat judge scores as a noisy, calibrated estimator, trended over time.

2

Drift is the early-warning signal

Watch judge-score distributions, guardrail pass rates, refusal rates, retry rates. A spike in PII detections or hallucination-fails tells you something changed before users report it — a model update, a prompt tweak, a shift in inputs.

3

The offline → online → guardrail pipeline

Offline evals gate the deploy; online evals monitor live; then production findings get promoted into real-time guardrails (Topic 8). For inline guardrails use small fast models, not full LLM judges — reserve those for the sampled/async layer. The whole loop from Topics 7–8, closed at cloud scale.

3.Cost: the line item that explodes

The counterintuitive fact of 2026: per-token prices fell ~280× from 2024, yet average enterprise AI spend rose ~483%. The units got cheaper; the units-per-task got dramatically more expensive — because agents do tool calls, retrieval, and multi-turn reasoning, and multi-agent fanout burns ~15× the tokens of a chat. If a task doesn’t genuinely decompose, you pay that multiplier without earning it — route those to a single-agent path.
a

Attribute at the trace root

Tag per-user / per-tenant / per-task at the root and propagate down, so every span inherits it and you can break spend down by customer, feature, or team. Observability tools estimate per-call cost from token counts × public pricing automatically.

b

Cardinality discipline

High-cardinality IDs (user_id, session_id) live in trace data, searchable — never as metric tags, or your cost metrics explode. Metric tags are for bounded dimensions: model, env, service, error type.

c

Route and cache for cost

Cheap/local model for simple prompts, frontier model for hard ones, decided per request; plus semantic caching. Together they cut spend 40–70%. Budget alerts sit on the per-tenant cost metric. (Note: tracing + a 5% judge sample is itself a real line item — tail sampling keeps it sublinear.)

4.Deployment & reliability for non-deterministic fleets

1

Rainbow deployments — drain, don’t kill

You met this on 10.1: agents run for minutes to hours, so you keep N versions live, route new sessions to the new one, and let old versions drain to completion. Temporal’s worker versioning pins each run to the version it started on. Killing a pod with a 6-hour agent on it is the mistake.

2

Autoscale on queue depth; retry with idempotency

Scale on queue depth + response time, not CPU (KEDA, scale-to-zero). And every side-effecting tool call needs an idempotency key — retrying a payment without one double-charges. Enforce a retry/token budget so a stuck agent can’t loop-spend forever.

3

SLOs on distributions, not per-request

Same input → different trajectory, so a raw success-rate SLO is noise. Define SLOs on aggregate signals — p95 latency, cost-per-task, aggregate online-eval score, guardrail pass-rate — and alert on distribution shifts, spending error budget against those aggregates.

The distinctly-2026 problem: the model-deprecation treadmill. Providers retire model snapshots on ~12-month cycles (Claude Opus 4 / Sonnet 4 pins retired June 2026). Use an alias like claude-opus-4 and behavior changes silently under you; pin a snapshot and it hard-fails at retirement. The mature answer is a model registry mapping every call site → pinned snapshot → last eval date → owning team, plus running your Topic 7 eval suite against the next model nightly — so a forced migration is a config change, not a fire drill. Also worth knowing: ~60% of production agent failures trace to provider capacity limits, which is why the AI gateway (10.3) doubles as a cross-provider failover layer.

5.The tool landscape, one line each

LangSmith

End-to-end platform, best for LangChain/LangGraph teams; closed source; also sells the deploy runtime from 10.4.

Langfuse

The leading genuinely-OSS/self-host option; OTel-native (acquired by ClickHouse, Jan 2026).

Braintrust

Evaluation-first, quality-as-engineering; trace-to-test, CI-native, auto-optimization.

Arize / Phoenix

Enterprise ML+LLM monitoring; Phoenix is OSS and OTel-native; strongest drift detection.

Datadog / AgentOps

Folds tokens + cost + traces into the APM dashboards you already run. Best if you’re on Datadog.

MLflow

OSS; ties agent tracing/eval into an existing ML lifecycle. For teams standardized on MLflow.

Build-vs-buy, cleanly: own the instrumentation (OTel GenAI conventions, so you’re not locked to a vendor SDK); buy the backend (storage, dashboards, eval UI, alerting). Self-host Langfuse/Phoenix when data residency or trace-volume cost demands it; buy SaaS when you’d rather not run ClickHouse and Kubernetes yourself.

6.What to monitor first, and what breaks

In priority order: (1) cost per task + token usage (the exploding line item, and the 15× multiplier bites first), (2) online eval score + guardrail pass-rate, (3) tool-call error/retry rate, (4) p95 latency + time-to-first-token. Everything else is secondary.

The agent-specific mistakes that recur in production:

Judge on the critical path

Running the LLM judge synchronously doubles latency and cost. Always async, always sampled.

Head-based sampling

Silently drops the rare, high-value failures you most need to see.

Retries without idempotency

Duplicate charges and writes. Every side-effecting call needs a key.

Killing agents on deploy

No rainbow/version pinning means mid-flight sessions die. Drain them.

High-cardinality metric tags

user_id/session_id as metric tags explodes cardinality. Keep them in trace data.

Trusting the offline suite as truth

It diverges from live behavior. You need online eval + drift monitoring.

7.Try it — read your own trace and budget

Exercise · ~10 minutes, in Claude Code
  1. You have a live example: the Account & Usage panel in Claude Code is production-ops telemetry for your own agent session. Open it and read what’s driving your usage — it literally attributes cost by contributing factor (subagent-heavy sessions, >150k context). That’s section 3’s cost attribution, applied to you.
  2. Spot the 15× multiplier in the wild: your heavy subagent use in this very project is the multi-agent token multiplier. Which of your tasks genuinely decomposed (earning it) vs which could have been single-agent?
  3. Design one alert: for a hosted version of your resume agent, write one SLO on a distribution (e.g. “p95 cost-per-publish < $X”) and one delta alert (“refusal rate jumps >2× week-over-week”). Notice neither is a per-request pass/fail.
  4. Trace it back: which earlier topics does this page operationalize? (Topic 1 loop = the spans; Topic 7 evals = the online layer; Topic 8 guardrails = the promotion target; 10.1 = rainbow deploys.)
