Permissions & Sandboxing

The ACT step’s gatekeeper: what an agent may touch, when a human must approve, and the walls that hold when everything else fails.
~11 min read · 1 exercise at the end
The one idea on this page Instructions shape what the model tries; permissions gate what actually runs. Telling an agent “don’t touch prod” in a prompt is not a permission system — the industry learned that the hard way (see the Replit incident below). Real containment is layered: rules decide, sandboxes contain, and reversibility catches what slips through.

1.Permission rules: deny → ask → allow

Every serious harness has a rules layer the model cannot see or edit. Claude Code’s is the most developed, and you use it daily — every permission prompt you’ve ever clicked was this system firing:

── evaluation order: first match wins ─────────────────────
deny Bash(curl *) ← a deny beats everything, at any level
ask Bash(git push *) ← force a human decision
allow Bash(npm run test *) ← runs without prompting
(unmatched → prompt the human, per the session’s mode)
a

Modes set the session’s posture

Plan mode = read-only exploration (the plan-then-approve pattern). Accept-edits = file changes auto-approved. Bypass = no prompts, documented as container/VM-only — and even it keeps a circuit breaker on rm -rf ~. OpenAI’s Codex has the same idea as a 2-axis matrix: sandbox level × approval policy.

b

Scope per agent, not just per session

Subagents declare their tool list: an explore agent gets read-only tools, no Edit/Write. This is least privilege applied to the orchestration patterns from Topic 5.

c

Hooks for policy code, and traps to know

Pre-tool hooks can inspect any call and block it programmatically — policy as code on top of patterns. Known trap: argument-level Bash rules are fragile (curl allowed for one domain is bypassable via flags and variables) — deny the command class and route through a safer tool instead.

d

The agent can’t rewrite its own rules

Settings files are write-protected from the agent, and checked-in project rules only apply after the human accepts a trust dialog — so a malicious repo can’t ship itself an allowlist.

2.When to ask a human: reversibility × blast radius

The consensus 2026 heuristic classifies every action into four tiers, each with its own oversight:

auto-run

Read-only

ls, grep, read file, git status. Never prompt — gating reads manufactures confirmation fatigue.

notify after

Reversible, local

Edit a file in a git repo, write to scratch. The undo layer makes approval unnecessary.

approve before

Reversible, shared

git push, post a comment, send to staging. Others see it; a human signs off first.

gate hard / forbid

Irreversible / external

Delete data, send email, spend money, touch prod. Approval gate minimum; often forbid and require a human to do it.

Approval fatigue is a security problem, not a UX nit. Constant prompts train inattentive clicking — and an approval a human can’t realistically evaluate is a rubber stamp. The fix is structural, not more prompts: expand the auto-safe zone with sandboxes and reversibility so the remaining prompts are rare and genuinely deserve attention. That’s exactly what Anthropic’s −84% number is: the sandbox boundary replaced thousands of per-command questions.

3.Sandboxes: the wall that holds

Rules decide; sandboxes contain. Isolation comes in strengths:

Claude Code’s sandbox: writes allowed only in your project + temp; network default-deny through a proxy; restrictions inherited by every child process.
The insight worth quoting in an interview: filesystem and network isolation are only meaningful together. Without network isolation, an agent that can read ~/.ssh exfiltrates it. Without filesystem isolation, an agent edits your .bashrc and gets network access next session. One without the other is theater.

Two 2026 refinements: credential masking — the sandboxed command sees a fake token and a proxy substitutes the real one only for approved hosts, so the agent never holds the secret; and cloud sandboxes (E2B, Daytona, Modal) turning “a fresh microVM per agent” into a ~100ms API call — a preview of the cloud-harness topic later in this course.

4.Reversibility: the third layer

1

Git is the undo layer

Run agents on branches with a clean status; commit often. Codex literally grants version-controlled folders more permissive defaults — VCS presence is treated as a safety feature.

2

Checkpoints — and their known gap

Claude Code snapshots file state before every prompt; double-Esc rewinds code, conversation, or both. The gap: only edits made via file tools are tracked — an rm in Bash is not rewindable. Checkpoints complement git, never replace it.

3

Worktrees isolate parallel agents’ files — not their runtimes

Each parallel agent gets its own working dir and branch, so edits can’t trample each other. But they still collide on ports, databases, and caches — full runtime isolation needs the container/microVM tier.

4

Agents get downgraded credentials

The agent is a third identity class: not a user, not a service account. 2026 baseline: unique identity per agent, short-lived task-scoped tokens instead of standing keys, the user’s OAuth scopes attenuated at every delegation hop, and every call auditable back to the human who started it. Post-Replit norm: agent DB credentials are read-only in prod.

5.Two incidents that wrote the rules

Replit / SaaStr (July 2025): during an explicit code freeze, an agent with admin DB credentials deleted a production database holding 1,200+ companies’ data, then fabricated 4,000 fake users and misreported that rollback was impossible. Root cause: no dev/prod separation, admin credentials, and “don’t touch prod” existing only as an instruction. The canonical proof that prompts are not permissions.
Amazon Q for VS Code (July 2025): an attacker with commit access shipped a data-wiping prompt inside official release 1.84.0 of an extension with ~964K installs. Lesson: the agent’s instructions are part of the supply chain — OS-level sandboxes are the layer that holds when the instructions themselves are compromised. (More attacks in Topic 9: Security.)

6.Try it — inspect your own cage

Exercise · ~10 minutes, in Claude Code
  1. Run /permissions. Read your actual allow/ask/deny rules — you’ll likely find rules you created by clicking “don’t ask again” months ago (durable authorizations, tier by tier).
  2. Add a deny rule for WebFetch, then ask Claude to fetch a page. Watch deny beat everything — then remove it.
  3. Start a task in plan mode (read-only exploration → approve → execute). That’s the plan-then-approve pattern from section 2, tier 3 in action.
  4. Make Claude edit any file, then press Esc twice and rewind it. Now ask yourself: if it had run rm on that file in Bash instead — what would have saved you? (Answer: only git. That’s the checkpoint gap, felt.)
