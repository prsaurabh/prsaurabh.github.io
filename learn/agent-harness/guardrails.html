1.Four checkpoints on the loop

Remember the loop from Topic 1? Guardrails are checkpoints bolted onto it — and an agent crosses them dozens of times per task, which is why their latency discipline matters far more than in a chatbot:

Input rails, action rails, output rails — plus a post-tool check sanitizing what re-enters the context.
in

Input rails

Topic/relevance filters (“is this a support question?”), PII masking before the text reaches the model or your logs, length and language checks, jailbreak screening (a pointer to Topic 9).

act

Action rails — the ones that bound real damage

Validate tool arguments against schemas and preconditions (amount ≤ invoice.balance), enforce spend caps and rate limits outside the model, cap loop iterations. The cheapest big win in the field: schema validation + idempotency alone bought 6–9 points of task completion.

out

Output rails

Schema/format validation, groundedness (“does the answer stay within the retrieved context?”), toxicity/brand checks, PII redaction on the way out — the model can leak PII it saw in context even when the input was clean.

2.The cost ladder: regex → classifier → judge

Every check sits somewhere on a three-rung ladder, and the universal rule is order checks cheap-to-expensive:

1

Deterministic (<1ms) — always on, every step

Regex, schema checks, allowlists, budget counters. Better yet, make the check unnecessary: structured outputs constrain decoding so invalid JSON can’t be generated at all — though valid schema ≠ correct values (“structured hallucinations” still need a semantic check).

2

Small classifiers (~10–160ms) — at trust boundaries

Fine-tuned 0.6–8B models (Llama Guard, Qwen3Guard, ShieldGemma) for toxicity, PII, off-topic. The 2026 sweet spot for accuracy-per-dollar — the “$1 guardrails” approach in this page’s video list.

3

LLM-as-judge (1s+) — final output and high-stakes actions only

Groundedness, brand voice, policy nuance. Too slow to run inline on every lap — sample it, run it async, or reserve it for the moments that matter. And use a different model family than the generator: one injection can compromise a same-family judge and generator together.

Three decisions every guardrail needs: (1) parallel or blocking — input rails can run concurrently with the model call and cancel it on a trip (faster, but tokens burn before cancellation); output and action rails must block. (2) fail-open or fail-closed — when the guardrail itself times out, block for irreversible actions, allow-and-log for advisory checks. (3) what happens on failure — hard refusal, or retry-with-feedback: re-prompt the model with the specific error (“field date was not ISO-8601”), which fixes most validation failures in one extra lap.

3.In the harness: hooks, and the eval → guardrail pipeline

a

Hooks are the attachment points

Modern harnesses don’t hardcode guardrails — they expose hooks. Claude Code’s pre-tool hook receives the full tool-call JSON and can allow, deny with a reason (fed back to the model as its next observation), escalate to a human, or rewrite the input. A denied pre-tool call is a hard stop; the call never executes. Cost: ~100–300ms per tool use, generally worth it.

b

Evals get promoted into guardrails

The 2026 pipeline: prototype a metric as an offline eval (Topic 7) → validate it online, async, non-blocking → promote it to an inline guardrail once precision and latency are proven. Corollary: guardrails themselves need evals — versioned regression corpora and CI that fails when a guardrail’s catch rate drifts.

c

Costs multiply by step count

100ms in + 150ms out + two tool checks per lap, across 10 laps, is seconds per task — and paid guardrails bill per evaluation. Mitigations: deterministic checks everywhere, expensive checks only at trust boundaries, check deltas not the whole context, cache verdicts on repeated content.

4.The landscape, one line each

OpenAI (Agents SDK + package)

Tripwire guardrails as first-class SDK constructs; a pre-flight/input/output pipeline package with built-ins for PII, jailbreak, off-topic, hallucination.

Guardrails AI

Open-source validator framework: 60+ composable validators, plus the reask pattern (re-call the LLM with structured error feedback).

NVIDIA NeMo Guardrails

Programmable rails runtime with a dialog DSL (Colang); strongest at conversational topic control; runs rails in parallel to hide latency.

AWS Bedrock Guardrails

Managed policy layer usable on any model: content/topic/PII filters, grounding checks, and unique formal-logic Automated Reasoning checks (“up to 99%” in rule-bound domains).

Azure AI Content Safety

Managed classifiers: harm categories, Prompt Shields for injection, and groundedness detection that can auto-correct ungrounded spans.

Anthropic

Safety pushed into the model layer: constitutional classifiers (jailbreak success 86%→4.4%; overhead now ~1%); app-level control exposed via harness hooks rather than a guardrails product.

What to guard first, in order: tool-call validation + hard budget caps (bounds real damage) → structured outputs (free reliability) → PII in and out (legal exposure) → grounding if RAG-based → topic scope → toxicity/brand last unless consumer-facing.

5.The mistakes everyone makes

Guarding only the input

Tools are the real risk surface. An agent with clean inputs and unvalidated tool calls is unguarded where it matters.

Prompt instructions as guardrails

“Never call delete” is probabilistic. The Replit incident (Topic 6) is what that looks like at scale.

Same-family judge and generator

Correlated failure: one injection compromises both. Use a different model family for the checker.

LLM judges on every lap

Latency death. Judges belong at trust boundaries, sampled, or async.

Set-and-forget

A near-zero hit rate is dead weight paying latency for nothing; a sudden spike is an attack or a drifted prompt. Dashboard hit rates; alert on deltas.

Forgetting streaming

An output rail that only sees the full completion moderates nothing — the user already read the text. Check chunks as they stream.

6.Try it — build a real guardrail for your resume rules

Exercise · ~15 minutes, in Claude Code

Your resume rules (“no em-dashes”) live in CLAUDE.md — which makes them prompt instructions: probabilistic. Promote one to a deterministic guardrail:

  1. Ask Claude Code: “add a PreToolUse hook that blocks any Edit or Write to files under pdf-resume/ if the new content contains an em-dash (—), with a deny reason explaining the rule.”
  2. Test it: ask Claude to insert an em-dash into resume-draft.html. Watch the hard stop — and watch the deny reason come back as the model’s next observation (retry-with-feedback, live).
  3. Test the false-positive side: make a legitimate edit with no dash and confirm it passes. That’s the FPR check from section 5, at n=1.
  4. Reflect: the rule existed in CLAUDE.md all along. What class of failure did you just eliminate, and which checkpoint (1, 2, or 3 from the diagram) does your hook sit at?
