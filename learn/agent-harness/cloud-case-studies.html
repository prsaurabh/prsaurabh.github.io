Published Case Studies

Six systems whose builders published the real engineering: the architecture, the numbers, what broke, and the lesson. This is the whole course — loop, context, tools, orchestration, sandboxes, evals, ops — seen through production deployments.
~12 min read · 1 exercise at the end
The one idea on this page Read enough production writeups and the same primitives recur, independently discovered by every team: durable execution for long runs, an orchestrator over specialized workers, a portfolio of models rather than one, deterministic guardrails wrapping probabilistic reasoning, and evaluation by replayable simulation. If your design has all five, you’re in good company. If it’s missing one, one of these teams already learned why the hard way.

1.Anthropic — Multi-Agent Research System

Claude “Research”

orchestrator-worker

A lead agent decomposes an open-ended query and searches the web, Workspace, and MCP integrations. The canonical published orchestrator-worker writeup.

Architecture
Opus 4 lead plans + saves the plan to memory (to survive 200K-token truncation), spawns 3–5 Sonnet 4 subagents in parallel, each running 3+ tools in parallel; a separate Citation Agent maps claims to sources. Subagents run synchronously — an acknowledged limitation.
+90.2% vs single-agent~15× chat tokens80% of variance = token spend−90% research time from parallelism
What broke: early agents spawned 50 subagents for simple queries, hunted for nonexistent sources, and duplicated work when delegation was vague. Fixed by encoding effort budgets in prompts and specifying objective/format/tools/boundaries per subagent.
Lesson: token budget is the dominant performance lever — scale agents and tool-calls to task complexity, in the prompt. Persist state to external memory, not the orchestrator’s context. (Topics 2b + 5.)
anthropic.com/engineering/built-multi-agent-research-system

2.LinkedIn — Hiring Assistant

Recruiter sourcing agent

plan-and-execute

LinkedIn’s first customer-facing agent: autonomously sources, evaluates, and engages candidates across 1.2B+ member profiles.

Architecture
Planner produces a structured task plan, Executor runs it — not a plain ReAct loop. A supervisor orchestrates 7 specialized sub-agents. Every instance has its own identity + mailbox; the whole lifecycle runs through async message passing on LinkedIn’s existing messaging infra. Per-recruiter cognitive memory, never used to train the LLM.
1.2B+ profiles searchedfine-tuned LLMs + speculative decoding
What broke: plain ReAct was unreliable at enterprise scale — models don’t follow complex instructions consistently, loops add latency, hallucinations kill recruiter trust. Plan-and-execute traded flexibility for reliability and fewer LLM calls.
Lesson: separate planning from execution to bound error-prone reasoning; give each agent its own instance + mailbox so memory can’t cross-contaminate customers; reuse battle-tested async messaging as the orchestration layer instead of inventing new coordination.
linkedin.com/blog/engineering · infoq.com/news/2025/09/linkedin-multi-agent

3.Cursor — Cloud (Background) Coding Agents

Autonomous cloud coding agents

durable execution

Coding agents that run unattended on dedicated cloud VMs, in parallel, for tasks spanning days. 40%+ of Cursor’s own PRs now come from them.

Architecture
Each agent on its own VM. Orchestration went from a home-grown work-stealing design (fragile, one 9 of reliability) to Temporal durable execution — surviving inference blips, pod hibernation, and week-long runs. They also shifted from one “eternal” agent workflow to many short workflows that exit after one task.
1→2 nines reliability from Temporal>50M actions/day>7M unique workflows
What broke: failures are silent — not a crash, just a subtle degradation in output quality. And a stuck cloud agent sits idle for hours unnoticed, so cloud agents need more-autonomous prompting than local ones.
Lesson: durable execution is the right primitive for long cloud runs (Topic 10.1); prefer many short workflows over one eternal loop; invest disproportionately in VM snapshot/fork/hibernate and environment fidelity — “the dev environment is the product.” Monitor quality, not just crashes.
cursor.com/blog/cloud-agent-lessons

4.Cursor — Composer (an agent model trained by RL)

Frontier agentic coding model

harness = RL environment

Cursor’s in-house agent model, RL-tuned for fast agentic coding. The case where the harness becomes the training environment.

Architecture
A Mixture-of-Experts LLM RL-trained inside the real agent harness — same tools (read/edit, search, grep, terminal), same sandbox. Training on PyTorch + Ray for async RL, MXFP8 MoE kernels. Evaluated on “Cursor Bench,” built from real engineer requests with hand-curated solutions.
~4× faster generation100Ks concurrent sandboxes1000s of GPUs
What broke: to make RL reflect real use, the reward environment had to be the actual harness — synthetic benchmarks optimize the wrong thing. That required a cloud fleet of hundreds of thousands of ephemeral isolated sandboxes.
Lesson: if you RL-tune an agent model, your cloud sandbox harness is your RL environment — its fidelity and parallelism (Topic 10.2) directly bound model quality. Speed is a first-class product axis for interactive agents.
cursor.com/blog/composer

5.Sierra — Enterprise CX Agents

Customer-experience agents

model portfolio + supervisors

Conversational agents for Sonos, WeightWatchers, SiriusXM — millions of consumers/month, doing recommendations, auth, returns, troubleshooting, churn prevention.

Architecture
A “constellation of models” — 15+ frontier, open-weight, and proprietary models, each chosen per task (fast models for lookups, classifiers for routing, long-context for policies, tone-tuned for conversation). High-agency tasks wrapped by supervisors enforcing guardrails; sensitive actions run through deterministic guardrails the agent cannot cross. Automatic cross-provider failover on outages.
15+ models routed per taskmillions of consumers/mo1000s of parallel regression sims
What broke: one model can’t serve all tasks — “models that shine at reasoning degrade when forced to answer quickly.” Latency-vs-reasoning is solved by routing, not by one compromise model.
Lesson: route per-task to a portfolio (fast / reasoning / tone) with cross-provider failover; wrap high-agency steps in deterministic supervisors (Topic 8); turn annotated production conversations into a large replayable simulation suite (Topic 7).
sierra.ai/blog/constellation-of-models · /agent-development-life-cycle

6.Intercom — Fin AI Agent

Autonomous support agent

pipeline, not a loop

Resolves support tickets across millions of conversations, grounded strictly in each customer’s own knowledge base.

Architecture
A 7-phase sequential pipeline, not an open agent loop: query refinement → retrieval (proprietary retriever) → rerank (proprietary reranker) → generation (purpose-built Fin Apex model, which answers or hands off) → a validation layer checking accuracy/safety before delivery → tuning → OWASP LLM Top-10 security. Answers constrained to the customer’s content, with an explicit “didn’t find it → escalate” branch.
76% avg resolution (85%+ for many)−65% hallucinations vs Sonnet 4.699.97% uptime
What broke: their stated failure mode of generic agents — “a thin layer on a general LLM creates a tradeoff between resolution rate and hallucination risk.” Fin engineers those as separate stages instead of trusting one prompt.
Lesson: for grounded, high-trust support, a deterministic retrieve→rerank→generate→validate pipeline with an explicit decline-and-escalate branch beats an unconstrained loop; a separate validation model gating output is where hallucination risk is actually controlled.
fin.ai/ai-engine · intercom.com/help (Fin outcomes)

7.Uber — Enterprise Agent Platform

1,500+ agents across 10,000 services

the boring plumbing

The case study for what a giant company must build before agents can run: a gateway, a registry, and an identity system for 60,000+ agent tasks/week.

Architecture
An MCP gateway auto-converts 10,000+ services’ protobuf/thrift IDLs into MCP tools with LLM-generated descriptions, shipped via version-controlled PRs through a central registry. Guardrails at the gateway: authz, PII redaction on responses, a hard block on mutable endpoints of critical services, two-tier internal/third-party trust.
Identity
Agents broke the human-vs-workload dichotomy. Every agent gets a SPIFFE identity; a token service mints single-hop, minutes-TTL JWTs embedding the full actor chain ([user, oncall-agent, investigation-agent]) so downstream systems see complete provenance. Built into the SDK so the secure path is the easy path.
60K+ tasks/week~1,800 code changes/week (~8%)<40ms p99 token exchange21K dev-hours saved (AutoCover)
Lesson: at enterprise scale the harness is the gateway + registry + identity system (Topic 10.3), and it takes longer than the agent loop. The three-identities-in-a-call-chain problem is real — solve it with actor-chain tokens, in the SDK, not a bolt-on proxy.
uber.com/blog/solving-the-agent-identity-crisis · aaif.io (Uber 60K tasks/week)

8.Shopify — Sidekick

Merchant assistant, millions of stores

single-agent + judge-as-reward

A consumer-scale merchant agent, and the best published eval-and-training case study — deliberately kept single-agent.

Architecture
One agent running the loop; multi-agent deliberately avoided (“single-agent systems handle more complexity than you’d expect”). Their signature pattern is JIT instructions: instead of a monolithic system prompt, each tool returns its usage rules alongside its output, exactly when needed — keeping the prompt lean past 50 tools and preserving the cache.
Evals + RL
LLM judges validated to Cohen’s Kappa 0.61 (vs 0.69 human) via a Turing-style swap test, on ground-truth sets sampled from real production. Then the judge became the RL reward (GRPO with gated rewards): syntax validity ~93% → ~99%. They published a reward-hacking taxonomy — opt-out hacking (politely refusing instead of trying), tag hacking, hallucinated IDs.
0.02→0.61 judge Kappa~93→99% syntax validity0–20 / 20–50 / 50+ tool-count phases
What broke: past ~50 tools, a monolithic prompt becomes “death by a thousand instructions” — hence JIT. And RL against a judge invites reward hacking, which must be caught explicitly.
Lesson: stay single-agent longer than you think (Topic 5); deliver tool instructions just-in-time to keep context lean (Topics 2 + 4); validate your judge against humans before trusting it, and expect reward hacking when the judge becomes a reward (Topic 7).
shopify.engineering/building-production-ready-agentic-systems · /fine-tuning-agent-shopify-flow

9.The five primitives that recur

Six independent teams, one convergent design vocabulary — and it’s exactly this course:

1

Durable execution for long runs

Cursor→Temporal, Anthropic checkpoints + rainbow deploys, LinkedIn async mailboxes. Prefer many short workflows over one eternal loop. (Topic 10.1)

2

Orchestrator + specialized workers

Anthropic 3–5 subagents, LinkedIn 7, Sierra supervisors — with explicit per-worker task contracts and token/effort budgets in the prompt. (Topic 5)

3

A portfolio of models, not one

Opus-lead/Sonnet-workers, Sierra’s 15+ routed models with failover, purpose-built fine-tunes (LinkedIn, Fin), a custom RL’d MoE (Composer). (Topics 3 + 10.5)

4

Deterministic guardrails around probabilistic reasoning

Sierra’s uncrossable guardrails, Fin’s validation gate, LinkedIn’s product-policy rails. The model reasons; code enforces. (Topics 6 + 8)

5

Evaluation by replayable simulation

Anthropic’s prod-prompt sims, Sierra’s conversation tests, Cursor Bench from real requests. Watch silent quality degradation, not just crashes. (Topic 7)

The through-line: nobody in this list bet everything on a smarter model. They won with harness engineering — the exact discipline this course is about. The model is the brain; everything you’ve learned across ten topics is the body that makes it useful in production.

10.Try it — audit your design against the primitives

Exercise · ~15 minutes, thinking + reading
  1. Score your own resume-agent design against the five primitives in section 7. Which do you have, and which would you add first if it became a real product? (Most hobby agents have zero of them — that’s fine, and knowing which to add first is the skill.)
  2. Pick the case study closest to something you’d build and read the primary source end to end (the Anthropic and Cursor writeups are the richest). Note one decision you’d have gotten wrong.
  3. Spot the anti-pattern you now recognize: which of these teams started with a plain agent loop and moved away from it, and why? (LinkedIn ReAct→plan-execute; Intercom loop→pipeline — both traded flexibility for reliability.)
  4. Close the loop on the course: name the topic each of the five recurring primitives came from. If you can, you’ve got the whole map.
