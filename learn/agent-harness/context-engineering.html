Context Engineering

The model has no memory — only a window. Deciding what fills that window each lap is the hardest and highest-leverage problem in harness design.
~8 min read · 1 exercise at the end
The one idea on this page Remember from Topic 1: every lap, the model re-reads the entire transcript from scratch. That transcript lives in a fixed-size context window (roughly a few hundred pages of text). Context engineering is the discipline of deciding, lap by lap, what deserves to be in that window — because everything in it costs money, and the wrong stuff in it makes the model dumber, not smarter.

1.What’s actually in the window

Picture the window as a jar that refills every lap. Some contents are fixed; the rest is under your control as the harness designer:

The system prompt and request are small. Tool results are what blow the budget — one careless “read the whole log file” can eat a quarter of the jar.
Cost is per lap, not per task. Because the model re-reads everything each lap, a bloated transcript isn’t paid for once — it’s paid for on every remaining lap. Junk added on lap 3 of a 40-lap run gets re-read 37 times.

2.Why more context makes models worse

The naive intuition is “give the model everything, let it figure it out.” Practice says the opposite. The known failure modes:

Context rot

As the window fills, attention spreads thin and accuracy on everything drops — even the early, important parts.

Lost in the middle

Models recall the start and end of the window well, the middle poorly. Key facts buried mid-transcript get missed.

Context poisoning

One wrong observation (a stale file, a misleading error) stays in the window and keeps steering later laps off course.

Distraction

Irrelevant detail competes with relevant detail. A model reading 10 files to answer about 1 performs worse than one reading just the 1.

So the goal is not the biggest context — it’s the smallest set of high-signal tokens that lets the model make the right next move. That phrase is worth memorizing; it’s the field’s working definition of context engineering.

3.The four tools of the trade

Every serious harness (Claude Code included) uses the same four techniques to keep the jar lean:

1

Just-in-time retrieval — don’t pre-load, fetch

Instead of stuffing everything the agent might need up front, give it tools to pull what it does need, when it needs it: search the codebase, read one file, grep for a symbol. The window holds answers, not the whole library.

2

Compaction — summarize and keep going

When the window nears full, the harness pauses, has the model write a dense summary of the transcript (decisions made, files touched, what’s left), then restarts the loop with that summary in place of the raw history. Lossy but survivable — like a detective’s case notes replacing hours of interviews.

3

Subagent isolation — spend someone else’s window

Big exploratory work (search 200 files for a pattern) is delegated to a subagent with its own fresh window. It burns 100k tokens reading files, then returns a 3-line conclusion. The parent’s jar gains 3 lines, not 200 files. This is the #1 practical reason subagents exist.

4

External memory — write it down, outside the window

Notes files, to-do lists, memory directories: the agent writes durable facts to disk and re-reads them on demand. Anything on disk costs nothing until fetched. This is how agents survive across sessions and week-long tasks.

4.Watch it go wrong, then right

Same task — “why is checkout slow?” — run by a naive harness and a context-engineered one:

── naive harness ──────────────────────────────────────────
lap 1 ACT read_file("logs/app.log") ← 80k tokens of log
lap 2 ACT read_file("src/checkout/") ×12 files ← 60k more
lap 3 THINK window 90% full; model now re-reads 140k junk tokens
every lap; misses the slow query buried mid-log…

── context-engineered harness ─────────────────────────────
lap 1 ACT grep("timeout|slow", "logs/app.log") ← 40 matching lines
lap 2 ACT spawn_subagent("scan src/checkout, find DB calls")
OBSRV "One unindexed query in cart_total.js:88" ← 1 line back
lap 3 ACT read_file("cart_total.js", lines 70–100) ← 30 lines
lap 4 DONE "Unindexed query on cart_items; add index on user_id."

Same model, same tools, wildly different outcome — purely from what was allowed into the window. That’s why this topic, not model choice, is where senior engineers earn their keep.

5.Try it — see your own window

Exercise · ~10 minutes, in Claude Code

You’ll inspect a real context window, watch it fill, and watch two of the four techniques fire.

  1. In any repo, run /context. That’s the jar diagram, live: see how much the system prompt, tools, and messages each occupy.
  2. Ask something that forces heavy reading, e.g. “search this whole repo and summarize how the resume pipeline works.” If Claude Code spawns an Explore subagent, you’re watching technique 3: many files read over there, a short conclusion returned here.
  3. Run /context again. Compare the messages share against step 1.
  4. Run /compact — that’s technique 2 on demand. Read the summary it produces: notice what it chose to keep (decisions, paths) and what it threw away (raw file dumps).
