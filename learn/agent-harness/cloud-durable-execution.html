The Agent Loop as a Cloud Workload

The loop from Topic 1 was a thing that ran in one process for a few seconds. In the cloud it must survive crashes, deploys, and day-long pauses — without re-sending the email it already sent. That is the durable execution problem.
~11 min read · 1 exercise at the end
The one idea on this page A cloud agent run outlives the process running it. The machine will vanish mid-run — OOM, node eviction, a routine deploy — and naively restarting re-executes steps that already had side effects (a second email, a second charge). Durable execution solves this: journal every completed step, and on recovery replay the history, skipping finished work, so each expensive or side-effecting step happens once across any number of crashes.

1.Why the request/response model breaks

Every assumption behind a normal web request fails for an agent:

1

Duration

A background coding agent clones a repo, runs tests, iterates, opens a PR — minutes to hours. Long-horizon agents hold state for weeks. No HTTP request or single serverless invocation lives that long.

2

Crashes and deploys

The environment will vanish mid-run. In-memory state — message history, loop counter, the pending tool result — is gone. A normal system would just 500 and let the client retry the whole thing. Here “the whole thing” is an hour of LLM spend.

3

Human-in-the-loop pauses

The agent proposes a refund or a prod change and must stop and wait for approval — possibly for days. A blocked process holding state open for a day is not viable.

The crux is side effects. Agent steps are not pure functions — they charge cards, send emails, merge PRs. So “just retry from the top” sends the email twice. The durability layer must give exactly-once replay of decisions, and turn the at-least-once reality of external calls into effectively-once with idempotency keys (“did this charge already happen?” checked against the journal before acting). And because tokens are the expensive part, you want to pay for each one once, across a dozen crashes.

2.Durable execution, explained simply

The runtime writes every completed step to a persistent log. After any failure it re-runs your code from the top — but each already-finished step returns its saved result instead of running again. Fast-forward through history, then continue live from the first un-run step:

── run starts ─────────────────────────────────────────────
step 1 plan = llm.complete(...) ✓ journaled
step 2 files = read_repo() ✓ journaled
step 3 send_email(summary) ✓ journaled ← side effect, done once
💥 CRASH — machine evicted here
── engine reschedules, replays from top ───────────────────
step 1 plan → returned from journal (llm NOT called again)
step 2 files → returned from journal (repo NOT re-read)
step 3 email → returned from journal (email NOT re-sent)
step 4 open_pr(...) ▶ first un-run step — continues live
a

Steps/activities are the escape hatch

Anything that touches the world or is non-deterministic — an LLM call, an HTTP request, the clock, a random number — is wrapped as a step. The engine runs it once, records the result, and on replay feeds the recorded value back as data.

b

The workflow body must be deterministic

Replay reconstructs control flow by re-running your orchestration code and matching each step call to the recorded history. Branch on Math.random() or wall-clock time directly and the replay diverges — corruption. All non-determinism must flow through the engine.

c

Checkpointing ≠ durable execution

LangGraph/CrewAI/ADK give you a save point but leave you to detect the failure, write the resume logic, and prevent duplicate side effects. True durable execution (Temporal, Restate, DBOS) does failure detection, self-recovery, and replay at the runtime level. Know which one you have.

3.Three camps, one guarantee

All deliver durability; they differ in where the machinery lives:

Workflow engines — external orchestrator

Temporal (defined the category; ~$5B valuation Feb 2026), Restate (lighter, single binary), Inngest (serverless-first), Hatchet (Postgres), Cloudflare Workflows, AWS Step Functions. Your code runs on workers; the engine owns the journal and drives recovery. Best for high throughput and strict isolation.

In-process libraries — durability embedded

DBOS (a library + a Postgres string; the durability record and your DB writes commit in the same transaction), LangGraph checkpointing (persist per super-step by thread_id), framework persistence in OpenAI Agents SDK, Microsoft Agent Framework, Dapr. No separate cluster. Best for small teams, side effects in your own Postgres.

Actor model — the agent IS the object

Cloudflare Durable Objects / Agents SDK: each agent is a globally-addressable stateful actor with its own SQLite that hibernates when idle (zero cost) and wakes on a message, WebSocket, or scheduled alarm. Run millions, one per user. “Project Think” (Apr 2026) adds durable fibers & mid-run checkpoints on top.

4.The patterns this unlocks

1

Background / async agents

The motivating category, all ~1 year old: Devin, OpenAI Codex Cloud, Cursor Background Agents, Google Jules, Claude Code Remote Tasks (Mar 2026). Assign a task → the agent runs alone for minutes-to-hours in a cloud sandbox → returns a PR. Durability is what lets it survive the hour without you babysitting a live connection.

2

Durable human-in-the-loop pause

At an approval gate the agent records an interrupt in the journal and the process exits entirely — zero resources held. Days later the approval (a signal, webhook, or alarm) rehydrates the exact state and resumes. This is Topic 6’s approval gate, made cloud-native.

3

Hibernation → millions of near-free agents

The actor headline: idle agents cost nothing until an event wakes them; state persists in SQLite across hibernation. This is what makes “one durable agent per user” economically viable.

4

Serverless vs persistent tradeoff

Idle-heavy, many agents, unpredictable pauses → serverless/actor + hibernation (scale to zero, pay per use, cold-start on wake). High-throughput tight loops → persistent warm workers (lower per-step latency, but you pay for idle and must drain on deploy).

5.Gotchas that bite in production

!

Determinism constraints

Inside workflow code: no Date.now(), no Math.random(), no direct network or DB calls. All of it goes through engine APIs or into a journaled step — or replay diverges and the run corrupts. (You’ve met this before: it’s why this course’s workflow tool bans those same calls in its scripts.)

2

Versioning across deploys

A run that started on v1 may still be executing when you ship v2. If v2 changes the control-flow shape, replay against the v1 history breaks. Temporal’s answer: worker versioning + pinning — pinned runs finish on the version they started on; new code only takes new runs.

3

Rainbow deployments

Blue/green with many colors: new workers take new tasks while old workers drain in-flight runs to completion, then retire. This is the cloud reason the field cares about draining — you can’t just kill a pod with a 6-hour agent on it.

4

When you don’t need it

A plain retry loop is fine for a short, single-process, idempotent pipeline. Reach for durable execution when you have state loss across timeouts, long human pauses, dozens of compounding steps, or expensive steps you can’t re-run. Don’t stand up a Temporal cluster for a 3-step, 5-second job — and mind the cost: engines bill per state transition, so don’t journal every token.

6.Try it — feel the replay

Exercise · ~10 minutes, thinking + Claude Code
  1. You’ve already used a durable engine: this course’s own workflow tool resumes a killed run from the last cached agent call. Ask Claude Code to explain how its workflow resumeFromRunId works — that’s journaling + replay, exactly section 2.
  2. Design one on paper: sketch a “publish my resume” cloud agent as durable steps — regenerate_pdf, sync_to_onedrive, commit_to_site. Which step is the dangerous side effect that must be idempotent? (The commit/push.)
  3. Break it in your head: the machine dies right after sync_to_onedrive. With durable execution, which steps re-run on recovery and which return from the journal? (Only commit runs; the first two replay from cache.)
  4. Decide: does this 3-step job actually need Temporal, or would a retry loop do? (Retry loop — it’s short and the steps are cheap. Now you know the boundary from section 5.)
