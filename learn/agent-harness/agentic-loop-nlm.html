The Agentic Loop: How an AI Agent Actually Works

What this is: a plain-language explanation of the single most important idea in AI agents, written to be read aloud and turned into a short explainer. Audience: a senior software engineer who is new to how agents work internally and wants to speak about them credibly. Goal: by the end, the listener should be able to explain what an agent really is, watch one run in their mind, and know why this loop is the foundation everything else is built on. Tone: clear, concrete, a little vivid — think a short documentary that demystifies a piece of technology.

Here is the one idea worth carrying out of this whole subject. An AI agent is not a smarter model. An agent is a plain, almost dumb loop wrapped around a smart model. The loop asks the model a simple question over and over — “what should we do next?” — carries out whatever the model asks for, shows the model what happened, and repeats until the model says it is finished. The loop is simple. The model is intelligent. And all of the real engineering lives in the wrapper around them. That wrapper has a name: the harness. This is a story about the harness.

Picture the loop as a circle that turns

Imagine a signal traveling around a circle, passing through four stations, again and again, until the work is done. Picture it glowing as it moves.

At the top of the circle is the first station: Think. This is the only station with any intelligence in it. The harness hands the model everything that has happened so far — the original request, the list of tools available, and every result gathered until now — and the model decides on a single next move. It either asks to use a tool, such as “run the tests” or “read this file,” or it declares that it is done and gives a final answer. Nothing is executed here. The model only ever asks.

The signal moves to the second station, on the right: Act. Here the harness — ordinary computer code, with no intelligence of its own — actually carries out what the model requested. It runs the command, reads the file, calls the outside service. Think of the model as a brain that can only speak, and the harness as the pair of hands that can actually touch the world. This station is also the security checkpoint: it is where the question “is this agent even allowed to do that?” gets asked and answered.

The signal continues to the bottom of the circle: the third station, Observe. The result of the action — the contents of the file, the output of the command, an error message — is written back into the running transcript of the conversation. This detail surprises people: the model does not remember anything between laps. It has no memory of its own. On every single lap it re-reads the entire transcript from the beginning. That transcript, growing lap by lap, is the agent’s memory. There is nothing else.

Finally the signal reaches the fourth station, on the left: Done? The check here is not mystical. It is simply this: did the model’s last reply contain a request to use a tool, or did it contain a finished answer? If it was an answer, the loop exits and the human gets their result. If it was a tool request, the signal travels around the circle again. And again. A single request might send the signal around the circle just once, or dozens of times — anywhere from one lap to fifty or more — before the work is truly finished.

That is the whole machine. Think, Act, Observe, check for done, repeat. Every AI agent product in the world — the coding assistants, the research agents, the customer-service bots — is this same circle wearing different clothes. If this pattern feels familiar to anyone who has studied AI, it has an older name: the ReAct pattern, short for Reason and Act, first described in 2022. Every modern agent is a descendant of that idea with better plumbing around it.

Watch one real run, lap by lap

Abstract diagrams only take you so far, so watch the loop do a real job — the kind of task a coding agent handles every day. A developer says: “the date test is failing, please fix it.” Follow the signal around the circle five times.

On the first lap, the model thinks: “I should see the failure before I do anything else.” It asks to run the test command. The harness runs it. The observation comes back: the test failed, it expected a properly formatted date but received the words “invalid date.”

On the second lap, the model thinks: “I need to read the code being tested.” It asks to open the source file. The harness opens it. The observation reveals the bug: the date-parsing function expects the day first, then the month, then the year — but the callers are handing it dates in the international year-first format.

On the third lap, the model has found it: “the format string is simply wrong.” It asks to edit the file, swapping the day-first format for the year-first one. The harness applies the edit. The observation confirms the change was made.

On the fourth lap — and this is the most important one — the model thinks: “never trust a fix; verify it.” Nobody instructed it to do this. It asks to run the tests again. The harness runs them. The observation comes back clean: all fourteen tests now pass.

On the fifth lap, the model thinks: “it is verified, there is nothing left to do,” and it gives its final answer, explaining that the parser expected the wrong date format and that the tests now pass. The loop exits. Notice the habit on that fourth lap — acting, and then verifying the act. A well-built agent closes its own loop that way, and that habit does not come from the model by accident. It comes from the written instructions the agent was given, which is a part the engineer writes by hand.

Who stops the loop, and why it matters

An endless loop of paid model calls is the nightmare that keeps harness designers awake, so a real harness has several ways to exit, ranging from graceful to blunt. The normal exit, which ends roughly nineteen out of twenty runs, is simply the model deciding it is done and replying with an answer instead of another tool request. A second exit is the human interrupting — pressing escape — and the harness handing control back. A third is a hard ceiling: a maximum number of laps, or a maximum amount of money spent, which stops a runaway agent from burning through a budget. A fourth is a refused permission, where the harness declines to run something dangerous, such as deleting files, and the model must either find another way or give up. And the last, bluntest exit is a fatal error — the outside service is down, or the memory is completely full — where the harness surfaces the problem rather than spinning forever.

There is one idea about stopping that deserves to be singled out, because it is most of the reason agents feel resilient rather than fragile. A failure is a lap, not a crash. When a tool errors — a file is missing, a test fails, a command breaks — the harness does not stop. It feeds that error back into the transcript as just another observation, and on the next lap the model reads the error and tries a different move. The agent stumbling and recovering, rather than falling over, is a direct consequence of this one design choice. Interviewers love this detail, because it separates people who have merely used an agent from people who understand how one works.

Why this one loop is the foundation of everything else

Once the loop is clear, every other topic in agent engineering reveals itself to be a knob on this exact machine, rather than a separate mystery. Managing what the transcript contains when it grows too large for the model to re-read is called context engineering — it is a knob on the Observe station. Writing the instructions the model reasons from is prompt engineering — a knob on the Think station. Shaping the menu of tools the model gets to choose from is tool design — a knob on the Act station. Running several of these loops at once and combining their answers is orchestration. Guarding what the Act station is allowed to touch is permissions and sandboxing. And grading whole loop runs to know whether they actually worked is called evals. Every one of these is a refinement of the same circle. That is why the loop is taught first: understand it, and the rest of the subject stops being a pile of buzzwords and becomes a single, understandable machine.

See it with your own eyes

The most convincing way to believe all of this is to watch it happen. Anyone using a coding agent like Claude Code can do a small experiment. First, ask it to create a simple sub-agent — for example, one that checks a piece of text against a short list of rules — and then open the file it creates. Inside, they will find three things: a list of the tools the agent is allowed to use, which is the Act menu; a written instruction that shapes how it thinks, which is the Think guidance; and, importantly, no traditional program code at all, because the loop itself lives inside the agent tool, not in this file. Then run it and watch the tool calls appear one at a time. Each one is a single lap of the loop described here — think, act, observe — made visible. Seeing the abstract circle turn into concrete steps on a screen is the moment the idea stops being theory and becomes obvious.

The five things to remember

  • An agent is a simple loop wrapped around a smart model; the model is the brain, the loop and its tools are the body, and the whole wrapper is called the harness.
  • The loop has four stations that repeat: Think, where the model decides one next move; Act, where the harness carries it out; Observe, where the result is recorded; and a check for whether the work is done.
  • The model has no memory of its own. It re-reads the entire growing transcript on every lap, and that transcript is the agent’s only memory.
  • A failure is a lap, not a crash: errors are fed back as observations, and the model adapts on the next lap. This is most of why agents feel resilient.
  • Every other topic in agent engineering — context, prompting, tools, orchestration, permissions, evals — is simply a knob on this one loop.
