Connectivity: Protocols, Identity, Discovery

Once agents live on different machines and in different orgs, they need to find each other, talk, and prove who they are — over the network, safely. By mid-2026 this settled into a clean two-layer stack.
~11 min read · 1 exercise at the end
The one idea on this page Two protocols, one sentence: MCP gives your agent hands; A2A gives your agents colleagues. MCP connects an agent to tools; A2A connects an agent to other agents across org boundaries. Both are now Linux Foundation projects, both fronted by gateways, both secured by OAuth 2.1 with a token bound to exactly one destination. That last rule is the whole ballgame for identity.

1.The two-layer stack

Real systems use both: A2A to delegate to a specialist, MCP for each agent to reach its own tools.

You met MCP on the Tool Design page as the “USB port for tools.” A2A is the layer above it, and the key distinction is what an agent exposes: through MCP an agent calls a capability; through A2A two agents collaborate without exposing their internal tools, prompts, or memory to each other. Governance is symmetric — both MCP and A2A are now hosted by the Linux Foundation’s Agentic AI Foundation (co-founded Dec 2025 by OpenAI, Anthropic, Google, Microsoft, AWS, Block), which is what makes “the stack” more than a marketing phrase.

2.Remote MCP: from subprocess to REST API

1

Two transports, split by locality

stdio for local dev (a subprocess over stdin/stdout). Streamable HTTP for anything remote — plain HTTP POST/GET, so an MCP server deploys and scales exactly like a REST API: behind load balancers, on K8s, serverless, or Cloudflare Workers.

2

The stateless turn (2026’s big shift)

Original MCP held a persistent session — hostile to horizontal scaling (sticky sessions, shared session stores). The 2026 spec reorients around a stateless core: independent requests, routing info lifted into HTTP headers, and app state made explicit — a tool mints a handle (basket_id) and the model passes it back as an argument, exactly like an HTTP cookie.

3

Registry + hosted offerings

The official MCP Registry (reverse-DNS namespaces, DNS/OAuth-verified ownership) curates ~2K servers against an ecosystem of ~18K. Enterprises run sub-registries (mirror upstream + internal tools + allowlists). Cloudflare, AWS AgentCore, and Azure all sell hosted MCP.

3.A2A: agents as colleagues

a

What it is

Two autonomous agents discover each other, delegate tasks, and stream results across org and framework boundaries. Transports in v1.0: JSON-RPC/HTTP, gRPC, REST, with SSE for streaming and webhooks for async push. Built by Google (Apr 2025), donated to the Linux Foundation (Jun 2025), stable v1.0 in early 2026.

b

Agent Cards — signed capability manifests

Discovery starts at a well-known path (/.well-known/agent-card.json): a machine-readable manifest of skills, endpoints, and auth requirements. v1.0 makes them cryptographically signed, so a consuming agent can vet a peer before delegating — the identity check A2A needs.

c

AP2 — agents that transact

An A2A extension for agent-driven commerce: signed “mandates” authorize transactions, backed by 60+ payments/financial orgs. The layer that lets agents pay, not just talk.

d

The protocol war ended in two, not three

IBM’s ACP overlapped heavily with A2A and folded into it (Aug 2025). So the stack is MCP + A2A — a nascent A2UI (agent↔UI) layer is discussed but not settled.

4.Gateways: the enforcement point

MCP and A2A define how to serialize a request but deliberately punt on the enterprise concerns. At scale you cannot let N agents each hold long-lived tool credentials and hit servers directly — so a gateway becomes the central control plane:

1

What it does

Terminates auth once, mints and binds short-lived downstream credentials, enforces per-tool RBAC and policy on every hop, applies rate limits and token budgets, and emits unified telemetry. It’s the load balancer + API gateway + policy engine, rebuilt for agent traffic.

2

The products

agentgateway (open-source Rust data plane, donated to the Agentic AI Foundation; native MCP + A2A, mTLS, OIDC, OPA policy per hop, OTel by default) is the reference. Plus TrueFoundry, Google Agent Gateway, and Azure API Management / API Center.

Why it’s mandatory at scale: without a gateway, every one of your agents is a credential holder and a policy blind spot. The gateway is where the Topic 9 defenses (least privilege, credential injection, audit) get enforced once, consistently, for the whole fleet — instead of hoping each agent got it right.

5.Identity: agents are a third class

Agents are neither human users nor static workloads — they need identities that are dynamic, short-lived, delegatable, and auditable. The mechanics:

!

Token audience separation — the non-negotiable rule

Under OAuth 2.1 (mandatory in the MCP spec since mid-2025), a token is bound to one specific server via its aud claim (RFC 8707). Server B must reject a token minted for server A. This is the primary defense against a compromised MCP server replaying your token elsewhere — the “confused deputy.” If you remember one thing from this page, it’s this.

a

Cross-app delegation — Okta XAA

Cross App Access implements an IETF identity-assertion grant so an agent can get a token for a third-party downstream API via the IdP both sides already trust for SSO — giving admins central visibility into agent→app connections instead of invisible OAuth grants. 25+ adopters including Anthropic, Atlassian, Cloudflare.

b

Cloud-native agent identities

Microsoft Entra Agent ID (one registry surfacing agents across Bedrock, Vertex, Databricks), AWS AgentCore Identity (a token vault storing scoped credentials for on-behalf-of access), Vertex agent identities bound to Google Cloud IAM. Common thread: short-lived scoped credentials + explicit delegation chains, replacing long-lived shared secrets.

The senior-interview nuance: three identities in one call chain. A single agent action carries three distinct identities that production systems must all handle: the human principal who asked (OAuth / Okta XAA), the agent software making the call (a client identity, e.g. a CIMD client-metadata URL), and the workload instance it runs on (a cryptographic identity via SPIFFE/SPIRE, mint-and-rotate ~hourly for service-to-service mTLS). The 2026 shorthand: SPIFFE for who the workload is, OAuth for what it’s allowed to do on whose behalf, OPA for policy. Conflating these is a common design mistake.

6.Discovery: how agents find each other

Three mechanisms, all converging on well-known JSON manifests: A2A Agent Cards (signed, for vetting peers), the official MCP Registry (verified server ownership), and Agentic Resource Discovery (ARD) — a draft where a site publishes an ai-catalog.json listing its agents, servers, and skills so registries can index them by natural-language capability.

Reality check on ARD: it’s the intended discovery layer for the “agentic web,” but still aspirational — a June 2026 census of 39 major sites (including every working-group member) found none actually serving a discoverable catalog. Signed Agent Cards and the MCP Registry are what work today; open natural-language discovery is a 2026 promise, not yet a reality. Worth knowing which is which in an interview.

How vetting works in practice today: fetch the well-known manifest → verify the signature (Agent Card) or namespace ownership (Registry) → read the declared auth requirements → obtain a correctly-audienced short-lived token → connect through a gateway that re-checks policy.

7.Where message buses fit

A2A’s transports (request/response + SSE + webhooks) are fine for orchestrator→specialist delegation, but not for high-fan-out or durable, replayable coordination. For those, production systems drop to an event bus: Kafka when agent events are part of the source of truth (durable, replayable), NATS/JetStream for low-latency fan-out (benchmarked at 1M+ msgs/sec). Rule of thumb: A2A/MCP for synchronous request and delegation; a message bus for asynchronous broadcast, buffering, and durable event flows across large fleets — with the gateway in front of both.

8.Try it — trace a cross-org call

Exercise · ~10 minutes, thinking + Claude Code
  1. You already use remote MCP: Claude Code connects to hosted MCP servers over exactly this stack. Run /mcp (or ask Claude to list its connected MCP servers) and note which are local (stdio) vs remote (HTTP).
  2. Draw the path: your orchestrator agent needs a specialist in another company to summarize a document. Which protocol connects the two agents (A2A), and which connects each agent to its own tools (MCP)?
  3. Find the audience rule’s teeth: if the specialist agent handed your token to a third service, what stops that service from accepting it? (The aud claim — the token names only its intended server.)
  4. Decide the transport: your fleet needs to broadcast one event to 5,000 agents and replay it later. A2A webhooks, or Kafka? (Kafka — durable, replayable fan-out is a bus job, not a request/response job.)
