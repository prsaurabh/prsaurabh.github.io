The Agentic Loop

The heartbeat of every agent. Once you see this loop, every agent product — Claude Code, Cursor, a LangGraph pipeline — becomes the same machine wearing different clothes.
~7 min read · 1 exercise at the end
The one idea on this page An agent is not a smarter model. It is a plain loop wrapped around a model: ask the model what to do next, do it, show the model what happened, repeat until the model says “done.” The loop is dumb. The model is smart. The engineering is in the wrapper — and that wrapper is called the harness.

1.The loop, drawn once

Everything happens in four repeating steps. The model never touches your files or runs commands itself — it only ever asks. The harness is the pair of hands.

One request, many laps. Each lap is one model call plus one tool execution.
1

THINK — the model call

The harness sends the model everything so far: your request, the tool list, and all previous results. The model replies with either a tool request (“run the tests”) or a final answer. This is the only step with intelligence in it.

2

ACT — the harness executes

The harness — ordinary code, no AI — actually runs the tool: reads the file, executes the command, calls the API. It also enforces permissions here: this is where “may this agent touch that?” gets checked.

3

OBSERVE — feed the result back

The tool’s output (file contents, test failures, an error) is appended to the conversation. The model doesn’t “remember” anything between laps — it re-reads the whole transcript every time. That transcript is the agent’s memory.

4

DONE? — the exit check

If the model responds with text instead of another tool request, the loop exits and you get the answer. Otherwise: another lap. Nothing mystical decides this — it’s literally “did the reply contain a tool call or not.”

You may know this as the ReAct pattern (Reason + Act, 2022). Every modern harness is a descendant of it with better plumbing.

2.Watch one real run

Here’s the loop running on a real task — the kind Claude Code does daily. Five laps, then done:

── you: "The date test is failing, fix it" ────────────────
lap 1 THINK "I should see the failure first"
ACT run_command("npm test")
OBSRV FAIL date.test.js — expected "2026-07-03", got "Invalid Date"
lap 2 THINK "Need to read the code under test"
ACT read_file("src/date.js")
OBSRV ...parseDate uses "DD-MM-YYYY" but callers pass ISO...
lap 3 THINK "Found it — format string is wrong"
ACT edit_file("src/date.js", "DD-MM-YYYY" → "YYYY-MM-DD")
OBSRV edit applied
lap 4 THINK "Never trust a fix — verify"
ACT run_command("npm test")
OBSRV PASS — 14/14 tests
lap 5 THINK "Verified. Nothing left to do."
DONE "Fixed: parseDate expected DD-MM-YYYY but received ISO dates. Tests pass."

Notice lap 4: nobody told it to re-run the tests. A well-prompted agent closes its own loop — act, then verify the act. That habit lives in the system prompt, which is exactly the part you write when you create a subagent.

3.Who stops the loop?

An infinite loop of paid model calls is the harness designer’s nightmare. Real harnesses have several exits, from best to worst:

Model says done

The normal exit: reply with text, no tool call. 95% of runs end here.

User interrupts

You hit Escape. The harness abandons the lap and hands control back.

Turn / budget cap

A hard ceiling: max laps or max tokens. Stops runaway agents from burning money.

Permission denied

Harness refuses a tool call (e.g. delete a file); the model must adapt or give up.

Fatal error

API down, context completely full. The harness surfaces the error instead of retrying forever.

Failure is a lap, not a crash. When a tool errors (“file not found”, “test failed”), the harness doesn’t stop — it feeds the error back as an observation, and the model tries a different move next lap. This one design choice is most of why agents feel resilient. Interviewers love this detail.

4.Why this matters for harness builders

Every knob you'll hear about in later topics is a knob on this loop:

5.Try it — watch the loop with your own eyes

Exercise · ~10 minutes, in Claude Code

You’ll create your first subagent (the resume rule checker) and then watch its loop run.

  1. In the interview-prep repo, tell Claude Code: “Create a subagent called resume-rule-checker that reads resume-writing-rules.md and checks any bullet I give it against every rule, returning pass/fail per rule. Give it read-only tools.”
  2. Open the file it creates in .claude/agents/. Find the three parts: the tool list (the ACT menu), the system prompt (the THINK instructions), and notice there is no code — the loop lives in Claude Code, not in this file.
  3. Run it: paste one bullet from your resume and ask for a check. Watch the tool calls appear one by one — each one is a lap of the loop you just learned.
  4. Spot the laps: which was THINK-heavy? Where did it OBSERVE the rules file? What made it decide it was DONE?
