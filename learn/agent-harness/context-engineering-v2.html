Context Engineering II: Production Scale

1.The cache changes how you write context

You met caching on the Prompt Engineering page. Manus — one of the most-watched agent teams — goes further: KV-cache hit rate is the single most important metric for a production agent. It reshapes three habits:

1

Append-only context

Never edit or reorder earlier transcript content — only add to the end. One changed byte invalidates the cache for everything after it. Even JSON key ordering must be deterministic.

2

Mask tools, don’t remove them

Tool definitions sit at the front of context; adding/removing tools mid-task nukes the whole cache (and confuses the model about tools it saw used earlier). Instead, keep the list stable and mask which tools are allowed right now. Naming helps: browser_*, shell_* prefixes let whole families be toggled.

3

Keep errors in the transcript

Counterintuitive: don’t clean up failed attempts. A stack trace left in context steers the model away from repeating the mistake. Erasing failures erases the evidence — the agent walks into the same wall twice. Error recovery is a hallmark of real agentic behavior.

4

Recite the plan (attention manipulation)

Manus constantly rewrites a todo.md and re-appends it — pushing the global plan into the model’s recent attention span to fight lost-in-the-middle drift on ~50-step tasks. Claude Code’s todo list does the same job. Cheap, and it works.

2.Memory: beyond the window entirely

Part 1 called this “external memory.” The 2026 landscape has real architecture behind it:

a

Memory files — embarrassingly effective

Plain markdown files (CLAUDE.md, NOTES.md) loaded at session start and edited by the agent itself. Anthropic ships this as an official memory tool. Benchmarks keep asking “is a filesystem all you need?” — and file-based memory keeps hanging with fancier systems.

b

Tiered memory (the MemGPT idea)

Treat context like RAM and external storage like disk: core memory (always in-window, editable), recall memory (searchable history), archival memory (vector store). Crucially, the agent itself pages data between tiers via tool calls — it curates its own memory.

c

Sleep-time consolidation

The frontier: a background agent “thinks” about memory while the system is idle — pre-computing inferences, reorganizing notes, distilling skills. Reported wins: ~5× less test-time compute for equal accuracy. The agent equivalent of what sleep does for you.

d

The honest caveat

Independent evals show memory layers fail in characteristic ways: retrieving the wrong memory, stale facts, loss during consolidation. When everything still fits in the window, full context beats most memory systems. Memory is for when it doesn’t fit.

3.Multi-agent context: the great debate

When several loops run at once, context becomes a coordination problem — and the field’s two most-cited essays disagree about it:

Cognition: “Don’t build multi-agents”

Every action carries implicit decisions. Parallel agents with different partial views make conflicting decisions and produce inconsistent work.

Share full traces or don’t split the work; prefer one linear agent with great compaction. Write-heavy work (coding) is their losing case for multi-agent.

Anthropic: orchestrator + workers won

Their research system — a lead agent farming out parallel subagents — beat a single-agent baseline by 90%. Read-heavy, parallelizable work (research) is the winning case.

The price: ~15× the tokens of chat, and token budget explained ~80% of performance variance. You buy quality with tokens.

By 2026 the vendors converged on a synthesis: one orchestrator owns the full conversation; subagents are ephemeral, isolated, and return compressed summaries. The craft is in the handoff:

The task brief is everything. Anthropic found vague briefs (“research the semiconductor shortage”) caused subagents to duplicate and drift. A proper brief has four parts: objective, output format, tool guidance, and explicit boundaries. And effort is scaled in the prompt: simple fact → 1 agent, few tool calls; complex research → 10+ subagents with divided lanes.

A quieter pattern is also back from the 1970s: the blackboard — agents coordinating through a shared workspace (usually just the filesystem) rather than messages. Rule of thumb, borrowed from Go: share memory by communicating (pass structured artifacts), don’t communicate by sharing memory (raw history dumps).

4.Surviving the long haul: compaction, for real

Part 1 introduced compaction as “summarize and keep going.” Here’s how Claude Code actually does it — the best-documented production example, in two layers:

── layer 1: microcompaction (continuous) ──────────────────
keep the ~5 most recent tool results, fully visible (the "hot tail")
clear older tool results → "[Old tool result content cleared]"
note everything cleared stays on disk, re-fetchable by path

── layer 2: full compaction (at threshold) ────────────────
write a structured summary: intent · decisions · files touched ·
errors & fixes · user messages · pending tasks · current work
keep the 5 most recently accessed files
re-add tool definitions, skills, CLAUDE.md (the stable prefix)
drop the raw transcript

The keep-vs-drop philosophy: preserve decisions, unresolved bugs, and intent; drop raw tool output (it’s reversible — the file is still on disk). Prefer reversible compression (replace content with a path or URL) before lossy summarization. And compact before the degradation zone, not when you hit the wall.

The human-in-the-loop version: the “frequent intentional compaction” workflow — Research → Plan → Implement, where a human reviews a written research doc and plan doc between phases, and each phase starts with a fresh window seeded from those docs. Teams report shipping a week’s work in a day on 300k-line codebases this way. You already do a version of this when you have Claude Code write a plan before coding.

5.What’s genuinely new in 2026

↗

Agentic retrieval demoted static RAG

The decision tree flipped: fits in context → put it in; agent can grep files → let it grep; only truly huge corpora → vector search. Claude Code’s grep/read loop is the exemplar. “Build a vector index” is no longer the default first move.

▣

Context compression, literally

Research frontier: rendering text as images for ~10× compression at 97% fidelity (optical compression), and sharing KV-caches between agents instead of text. Not production practice yet — watch this space.

∞

Million-token windows didn’t kill any of this

Context rot persists and prefill cost scales with tokens, so the effective quality window stays far below the advertised one. The framing just shifted: from “fit everything” to “minimize prefill compute without degrading quality.”

☰

The recipe standardized

Planning todo + filesystem + ephemeral subagents + memory files — the Claude Code shape — is now the generic “deep agent” stack, shipped as reusable harnesses by LangChain and others. Learn one, you’ve learned them all.

6.Try it — write a real task brief

Exercise · ~15 minutes, in Claude Code

The most transferable skill on this page is the four-part subagent brief. Practice it on your own repo:

  1. Write a brief for a subagent using all four parts — objective (“find every place my resume pipeline hard-codes a file path”), output format (“a table: file, line, path found”), tool guidance (“grep and read only, no edits”), boundaries (“only pdf-resume/ and scripts, skip archives”).
  2. Give it to Claude Code prefixed with: “delegate this to a subagent exactly as briefed.”
  3. When it returns, judge the handoff: did you get a compact table (good isolation — 1–2k tokens back) or a rambling dump? Did it stay inside the boundaries?
  4. Now deliberately re-run with a vague version (“check my repo for path issues”) and compare. You’ll see Anthropic’s duplication-and-drift finding first-hand.
