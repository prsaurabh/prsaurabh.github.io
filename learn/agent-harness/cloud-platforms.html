Managed Platforms: Buy the Bundle

Pages 10.1–10.3 taught the primitives — durable runtime, sandboxes, connectivity and identity — as things you wire yourself. This page is the shortcut: the hyperscalers now sell all of them pre-assembled. What each bundles, and how to choose.
~10 min read · 1 exercise at the end
The one idea on this page Wiring the last three pages together — a checkpoint store + a queue + a microVM pool + a secret vault + an OAuth broker + a trace pipeline + an autoscaler — is a systems-integration project on its own. Between Oct 2025 and Q1 2026 the three hyperscalers converged on the same answer: a managed bundle of runtime + memory + identity + sandboxed tools + gateway + observability, framework- and model-agnostic. The rule to remember: the bundle owns the substrate; you still own the agent’s brain and its tools.

1.What a bundle actually is

Every platform packages the same six things — and each maps directly to a page you’ve already read:

The bundle sells operations, not orchestration. None of them write your agent’s brain.

Why now: the primitives matured enough to standardize, and enterprises hit the wall where “our pilot works but we can’t operate 50 agents safely” — identity, audit, residency, and cost governance became the blockers, not the model. AWS and Google then published near-identical per-vCPU-hour rates, which is what a commoditizing runtime layer looks like.

Note the two escape hatches at the bottom: LangSmith (self-hostable) and Temporal (durable core only) are what you reach for when you want the runtime but not the lock-in. And watch the OpenAI deprecation — Agent Builder and Evals wind down Nov 2026; the Agents SDK is the durable path. Two naming catches for 2026: Google retired the “Vertex AI” brand in May 2026 (the stack is now the Gemini Enterprise Agent Platform, Agent Engine → “Deployments”), and Azure’s is now “Microsoft Foundry” — older docs use the old names.

The platform-risk exhibit: OpenAI launched AgentKit’s Agent Builder in Oct 2025 and deprecated it eight months later. When you pick a bundle, weigh vendor durability, not just today’s feature list — and note that MCP/A2A and framework-agnostic runtimes have shrunk lock-in to mostly your memory store, identity wiring, and observability history, not your code.

3.How to choose

The factors, in the order that actually decides it:

1

Where your data and cloud already live

This dominates. Co-locate the agent with the data it acts on: data in S3/Redshift → AgentCore; data gravity in BigQuery → Vertex; a Microsoft 365 / SharePoint / Teams shop → Foundry. Latency, residency, and IAM continuity all follow from this one choice.

2

M365 integration and model preference

If the agent lives inside Teams/Copilot, Foundry’s Entra identity + one-click publishing is uniquely strong. On models: AgentCore is the most agnostic, Vertex leans Gemini, Foundry leans OpenAI, AgentKit is effectively single-vendor.

3

Lock-in tolerance and cost shape

LangGraph-standardized and want to avoid a hyperscaler? LangSmith (self-host) or Temporal + your own primitives. On cost: idle-aware consumption (AgentCore) rewards bursty LLM-wait-heavy agents; Foundry’s “no runtime charge” is cheapest to start; steady 24/7 agents may be cheaper on owned infra.

4

Maturity vs cutting-edge

AgentCore (Oct 2025) is the most battle-tested with the longest window; Foundry hosted agents only GA’d Jul 2026. Weight this by how much you need proven vs newest.

The pattern big enterprises actually run: more than one. Foundry for internal M365 agents and Bedrock or Vertex for customer-facing product agents — because the choice is per-workload, not per-company. Nobody picks a single platform for everything.

4.Buy vs build

Buy

Buy the bundle when…

speed matters, you have a handful of platform engineers, you want identity/audit/residency solved by default, workloads are bursty (consumption wins), and you’re already on the matching cloud. It collapses months of substrate integration into a deploy command.

Build

Assemble primitives yourself when…

steady high-utilization agents are cheaper on owned infra, you need portability across clouds, you have custom isolation/GPU/networking needs the runtime won’t expose, or you already run Temporal/K8s so owning the runtime is cheap.

The pragmatic middle wins most of the time: buy the runtime + identity + observability (the tedious, security-critical parts), and build/own the tools and orchestration. That’s exactly why every bundle is sold as modular building blocks — most teams adopt Runtime + Identity + Observability and bring their own framework and MCP tools. This course, in fact, has taught you to build the “brain” half; the bundle is the “substrate” half.

5.Try it — spec a platform choice

Exercise · ~10 minutes, thinking
  1. Map your own use case: if you turned the resume-publish agent from 10.1 into a hosted product for others, which factor from section 3 decides your platform first? (Where the data lives — likely nowhere yet, so cost/lock-in and model-agnosticism lead.)
  2. Do the buy-vs-build call: it’s a low-volume, bursty, mostly-idle agent. Buy or build? (Buy — idle-aware consumption billing on a bundle beats paying for owned idle infra.)
  3. Draw the boundary: name one thing the bundle gives you and one thing you’d still write yourself. (Given: runtime, identity, sandbox, traces. Yours: the orchestration graph and the MCP tool that regenerates the PDF.)
  4. Spot the lock-in: which two options let you keep the runtime if you leave the cloud vendor? (LangSmith self-host and Temporal.)
