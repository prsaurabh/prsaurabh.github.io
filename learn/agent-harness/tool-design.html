Tool Design

Tools are the ACT menu from Topic 1. Designing them well — and daring to have fewer of them — moves agent success rates more than almost anything else you control.
~10 min read · 1 exercise at the end
The one idea on this page A tool is a contract between a deterministic system and a non-deterministic agent. You are not writing an API for developers — you are writing one for a model that will read the description, guess when to call it, and reason over whatever comes back. Design for that reader, and remember the 2026 surprise: removing tools often makes agents better.

1.Anatomy of a good tool

Same tool, written twice. Every difference below is a documented, measured practice:

✗ Written for a developer

name: search
desc: "Searches the DB."
params: q, u, mode
returns: raw JSON dump —
  uuids, mime types, 40 fields,
  unbounded list
on error: "ERR_422"

✓ Written for the model

name: contacts_search
desc: "Search saved contacts by
  name or email. Use before any
  send_message call. Not for
  company lookup: use org_search."
params: query, user_id,
  response_format: concise|detailed
returns: names + emails only
  (ids included only if needed downstream)
on error: "No match for 'jhon'.
  Try partial name, e.g. query='jo'"
1

Name and namespace deliberately

asana_search vs search: prefixes mark service boundaries and prevent confusion between similar tools. Anthropic found even prefix-vs-suffix choice measurably moves eval scores.

2

Describe it like onboarding a new teammate

Make the implicit explicit: data formats, when to use it, when not to, how it relates to other tools. Description refinements alone drove a state-of-the-art SWE-bench result. Adding concrete usage examples took parameter accuracy from 72% to 90%.

3

Design the response for reasoning

Return high-signal, human-readable fields; strip UUID soup and raw payload noise — models reason better over names than opaque IDs. Offer a concise|detailed knob (concise ran ~⅓ the tokens in Anthropic’s example). Truncate: Claude Code caps tool responses at 25K tokens.

4

Errors are recovery prompts

“ERR_422” wastes a lap (remember: failure is a lap, not a crash). A good error says what was wrong, shows a valid example, and points at the cheapest retry. You are prompting the next THINK step.

2.Fewer, better tools

The strongest 2026 finding is about count, not quality. Build tools around workflows the agent actually pursues, not around your API’s endpoints: one schedule_event that finds availability internally beats list_users + list_events + create_event chained across three laps.

The evidence: Vercel deleted 80% of its text-to-SQL agent’s tools, gave it a sandboxed bash instead — success went from 80% to 100%, 3.5× faster, ~40% fewer tokens. Their post-mortem line is the lesson: “we were constraining the model’s reasoning because we didn’t trust it to reason.” GitHub cut Copilot’s agent from 40+ tools to 13 and gained 2–5 SWE-bench points. OpenAI’s guidance: under ~20 tools exposed per turn.

Two refinements before you go deleting everything:

a

Overlap kills, not count

The real failure mode is two tools that plausibly do the same job — the model hesitates or picks wrong. A multi-system orchestrator can carry 20–30 distinct tools fine; a single-system agent usually wants 3–5.

b

Defer the long tail

Don’t load 50 tool definitions (~72K tokens) up front — load a ~500-token search tool and let the agent fetch definitions on demand. Anthropic measured 85% context savings AND large accuracy gains (Opus 4 on MCP evals: 49% → 74%). You’ve seen this live: it’s how Claude Code handles its own deferred tools.

3.The action-space ladder

The deepest 2026 shift: for capable models, the best “tool” is often not a tool at all. There’s a ladder of generality:

Climbing a rung trades auditability for generality and token efficiency.

Rung 3 is where the eye-popping numbers live. Instead of loading thousands of tool definitions and round-tripping every intermediate result through the model, the agent sees tools as a filesystem of code APIs, reads only the files it needs, and writes a script: Anthropic measured one workflow dropping from 150K tokens to 2K (−98.7%); Cloudflare’s Code Mode wrapped their entire 2,500-endpoint API for ~1K tokens. Bonus: sensitive intermediate data stays in the sandbox and never enters the model’s context.

4.MCP: the USB port for tools

The Model Context Protocol is how tools standardized. One protocol, and any agent can connect to any of ~10,000+ public servers (GitHub, Slack, databases…). It’s now vendor-neutral — Anthropic donated it to the Linux Foundation in late 2025, with OpenAI, Google, AWS, and Microsoft on board — and SDK downloads run ~97M/month. When you connected tools in Claude Code, MCP was underneath.

The security flip side (preview of Topic 6): a tool description is text the model trusts — so a malicious MCP server can embed instructions in one (“tool poisoning”). Combine untrusted input + sensitive data access + any channel out, and you have the “lethal trifecta” — the default shape of most agents. 30+ MCP CVEs were filed in Jan–Feb 2026 alone, and the NSA published MCP security guidance in June. Rule of thumb: treat installing an MCP server like installing a browser extension with your credentials.

5.Iterating: let the agent fix its own tools

Tool quality is discovered, not designed. The reference loop, from Anthropic’s tool-writing guide:

1

Test agent-in-the-loop, never just as functions

A tool that passes unit tests can still fail agents constantly. Wire it into a real loop and watch real tasks.

2

Read the transcripts

The universally-cited highest-yield practice: read full trajectories (including the model’s thinking) to see why a tool was misused — not just pass/fail counts. Measure tokens, laps, and error rates alongside accuracy.

3

Then hand the transcripts to the agent

Anthropic’s striking result: feeding eval transcripts to Claude and letting it rewrite the tool descriptions and implementations beat expert human versions on held-out tests. The model knows what confused it.

And the mistakes to avoid — each one is a failure someone shipped:

1:1 endpoint wrapping

Mirroring your REST API instead of building workflow tools. The original sin.

Overlapping vague tools

Two tools that might both fit → wrong picks and hesitation. The real “too many tools” bug.

Raw payload returns

UUID soup, 40 fields, unbounded lists. The agent pays tokens to ignore them — every remaining lap.

Opaque errors

“ERR_422” teaches nothing; the agent retries blind or gives up.

Assumed context

Formats and jargon the model can’t know. Write for the new teammate.

Over-constraining strong models

Rigid multi-tool pipelines where bash or code execution would do — Vercel’s lesson.

6.Try it — critique and rebuild a tool

Exercise · ~15 minutes, in Claude Code
  1. Ask Claude Code: “show me the definition of your Read tool — name, description, parameters — and critique it against these criteria: written for the model, unambiguous parameters, response design, recovery-oriented errors.” You’ll see a production tool definition and how it holds up.
  2. Now design one of your own on paper: a single workflow tool publish_resume that regenerates your resume PDF and syncs it to the website — one tool, not three. Write the description as if onboarding a teammate, add a version: short|full parameter, and write one good error message for “Playwright not installed.”
  3. Paste your definition into Claude Code and ask: “as the agent that would call this, what would confuse you?” — that’s step 3 of the iteration loop, agent-optimizing-tools, on your own work.
  4. Spot the ladder in the wild: notice that Claude Code mostly uses Bash (rung 2) rather than a hundred atomic tools — count the distinct tools it used in your last session.
