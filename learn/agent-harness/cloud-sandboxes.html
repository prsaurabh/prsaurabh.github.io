Sandboxes & Compute at Scale

Topic 6 put a sandbox around one agent on your laptop. This is the same idea turned into a fleet: where thousands of agents’ untrusted, model-written code runs as a hosted service — and how you keep it fast, isolated, and cheap.
~11 min read · 1 exercise at the end
The one idea on this page When an agent runs code it just wrote, the isolation boundary is the whole product. The entire market is one tradeoff played out at scale: stronger isolation vs faster cold start vs higher density (cheaper). Containers are fast but share the host kernel; microVMs give each sandbox its own kernel behind a hardware wall. Pick your point on that curve, then make idle sandboxes cost nothing with snapshot-and-resume.

1.The isolation ladder, at scale

Five models, arranged from “shared kernel, fast” to “own kernel, strong.” This is the Topic 6 ladder again, but now the rungs are products you rent:

The boundary that bites: a Docker sandbox escape affects every other sandbox on that host — they share one kernel. That’s the structural reason the container-based providers win on speed but carry a weaker isolation story than the microVM providers. And a surprise at scale: networking, not compute, is the real cold-start bottleneck — TAP/CNI setup can inflate startup latency by ~260%, and parallel starts degrade sharply past ~400 concurrent. The gold-standard projection: ~60% of Kubernetes sandboxed workloads on Firecracker by 2026.

3.The Kubernetes-native path

Since two-thirds of GenAI orgs already run Kubernetes, the agent-sandbox problem is being solved natively there — buy-vs-build hinges on whether you already live on K8s:

1

GKE Agent Sandbox

gVisor isolation (“the same tech that secures Gemini”), 300 sandboxes/sec/cluster, 90% of allocations under 200ms. Open-sourced as a Kubernetes SIG subproject — runs on any cluster. New CRDs: Sandbox, SandboxTemplate (the security blueprint), SandboxClaim, SandboxWarmPool.

2

Agent Substrate — for ultra-scale

A thin control plane above Kubernetes for 10s–100s of millions of agents. Standard K8s chokes on “the chatter of millions of sub-second tool calls,” so Substrate multiplexes agents on/off ready capacity in real time with data-locality-aware scheduling.

3

kagent — agents as CRDs

CNCF project: define agents declaratively, version them in Git, review in PRs, roll out via GitOps. Every agent is a pod. The Topic 5 orchestration you did in prompts, expressed as Kubernetes objects.

4

Buy vs build

Buy a service (E2B/Modal/Daytona/Vercel) for sub-200ms managed cold starts without owning microVM networking and warm pools. Run your own (GKE Agent Sandbox / Northflank BYOC) when you live on K8s, need data locality or compliance, or are at the “millions of agents” scale.

4.The capabilities that make agents economical

Snapshot → resume kills idle cost; snapshot → fork branches parallel exploration from one prepared base.
a

Snapshot / pause / resume

Capture full filesystem + memory (running processes, loaded state) and resume in ms. An idle agent stops billing yet restores instantly — the core lever for bursty workloads. This is the cloud-scale version of Topic 10.1’s hibernation.

b

Fork

Clone full execution state to branch parallel exploration from one prepared base — no repeated setup. This is what makes the tree-search / multi-attempt patterns from Topic 5 economical.

c

Default-deny egress

The norm now: per-sandbox allowlists in the template, plus credential injection via proxy so generated code never holds the real secret. This is Topic 9’s trifecta defense, enforced at the fleet level.

d

Warm pools, backed by suspended VMs

Pre-booted sandboxes hide cold-start latency; backing the pool with suspended VMs rather than live ones makes sub-200ms allocation affordable — idle cost traded for latency, cheaply.

Reality check — managed isolation is an approximation. AWS Bedrock AgentCore’s “no external access” sandbox mode was bypassed in late 2025 via DNS tunneling (encode data in subdomains of an attacker’s domain), and its Firecracker metadata service leaked IAM credentials over plain HTTP — an IMDSv1 rerun inside a microVM. AWS made the v2 metadata service mandatory in Feb 2026 and re-worded the docs from “no” to “limited” external access. Lesson (straight from Topic 9): managed sandboxes need egress paths for their own plumbing, so isolation claims are architectural approximations — layer defenses and monitor egress, don’t trust the label.

5.GPUs, and the boundary of this page

Two different GPU needs, only one of which lives here:

1

Agent-generated code that uses a GPU

The agent writes code that trains or runs a model. That needs a GPU inside the sandbox (Modal, Northflank, Daytona, Fly). Friction: GPU passthrough into microVMs is real engineering, and multi-tenant GPU sharing without hardware partitioning shows 55–145% latency degradation.

2

The agent’s own reasoning — out of scope

The LLM driving the loop almost always calls a hosted API (Anthropic/OpenAI/Google), not a self-hosted model, because the TCO favors it below high sustained volume. Self-hosting the inference-serving stack is an inference-infra topic, not a sandbox one. The only thing that matters here: the sandbox fleet stays CPU-dense and cheap because the model lives elsewhere.

6.The cost move that matters most

Scale-to-zero + snapshot is the whole game. An agent idle 95% of the time should hold zero live compute. Snapshot/suspend turns “keep the VM hot” into “restore in ms when a tool call arrives.” The corollary bites in production: providers whose working-memory snapshots aren’t GA yet force agents to run continuously at full rate just to hold state — a large hidden cost. And on billing, prefer providers that meter active CPU (Vercel Fluid, Cloudflare) over wall-clock for IO-bound agents that spend most of their time waiting on an LLM or a tool: up to 95% cheaper.

Density decides unit cost: containers/gVisor pack more sandboxes per host (cheaper each, weaker isolation); Firecracker’s ~5MB/VM overhead keeps microVM density viable while giving the hardware boundary. Choosing gVisor vs Firecracker is largely a density-vs-isolation cost curve. At the “millions of sandboxes” frontier the constraint stops being per-sandbox compute and becomes control-plane throughput — which is exactly why Agent Substrate exists.

7.Try it — place your workload on the curve

Exercise · ~10 minutes, thinking + Claude Code
  1. You’ve used a cloud sandbox already: this course’s Agent tool can run subagents with isolation: "worktree", and background/remote tasks run in exactly this kind of hosted sandbox. Ask Claude Code what isolation its background tasks use.
  2. Place a workload: for a fleet running untrusted user-submitted Python, which rung of the section-1 ladder do you pick, and why? (Firecracker — you can’t risk a shared-kernel escape across tenants.)
  3. Now flip it: an internal agent running your own trusted scripts, latency-critical, thousands per second. Which rung? (Containers or gVisor — trusted code lets you trade isolation for speed and density.)
  4. Cost check: your agent spends 90% of wall-clock waiting on LLM calls. Which billing model saves the most, and which capability keeps idle cost at zero? (Active-CPU billing; snapshot-and-resume.)
