Prompt Engineering

What still works, what reasoning models made obsolete, and how production harnesses version, test, and cache their prompts like code.
~9 min read · 1 exercise at the end
The one idea on this page Prompt engineering in 2026 is not wordsmithing magic phrases. It is eval-driven engineering: prompts are assembled from modular parts, versioned like code, gated by tests before deploy, and laid out to maximize cache hits. The golden rule for the words themselves: write for a brilliant new employee with zero context — if a colleague would be confused by your instructions, so will the model.

2.Techniques that still earn their keep

1

Clear, direct instructions — with the why

Numbered steps, explicit output format, explicit constraints. Giving the motivation (“output goes to a screen reader, so no tables”) beats bare orders — models generalize from reasons to cases you didn’t anticipate.

2

Few-shot examples

Still the most reliable lever for format and tone: 3–5 diverse examples in <example> tags, worth +15–30% on format-sensitive tasks. Start zero-shot; add examples only when consistency fails.

3

Structure with XML-style tags

Effectively mandatory in serious prompts: <instructions>, <context>, <documents> keep content types unambiguous. Claude Code’s own system prompt is heavily tag-structured.

4

Long-context layout

Big documents at the top, question and instructions at the end — up to ~30% better on multi-document tasks. Ask for supporting <quotes> before analysis to ground the answer.

5

Self-check endings

“Before finishing, verify your answer against [criteria]” — cheap, reliably catches code and math slips. The prompt-level version of lap 4 in Topic 1’s trace.

6

Prompt chaining — narrower now

Draft → critique → refine as separate calls survives where you need inspectable intermediate output. For general multi-step reasoning, modern models handle it internally — don’t chain what one call can do.

7

Meta-prompting — let the model write the prompt

Mainstream now: paste a failing prompt plus transcripts into a strong model and ask it to rewrite. Programmatic versions (DSPy, GEPA — an ICLR 2026 oral) evolve prompts against eval scores and often beat hand-written ones by ~8%.

3.What reasoning models made obsolete

Half the classic prompt-engineering folklore quietly died when models learned to reason natively. Knowing what to stop doing is now as valuable as the techniques themselves:

“Think step by step”

Reasoning models think natively; scripted CoT adds little. Instead: say “think thoroughly” and let the model plan — its own plan usually beats your hand-written steps.

Hand-rolled ReAct prompts

Writing “Thought:/Action:/Observation:” templates is dead. Instead: native tool calling + the harness loop from Topic 1 do this structurally.

Self-consistency voting

Sampling N answers and majority-voting is mostly displaced. Instead: turn up the effort / reasoning knob — the primary quality/cost lever now.

ALL-CAPS “you MUST” nagging

Actively harmful on current models — causes tool over-triggering. Instead: plain, calm statements; fewer instructions, better followed.

Emotional bribes

“This is important to my career” tips faded to trivia. Instead: state real stakes as real constraints if they matter.

Answer prefilling

Pre-writing the start of the reply to force format is deprecated (errors on new Claude models). Instead: structured outputs / JSON schema — format is an API feature now, not a trick.

Interview angle: “What prompting techniques would you remove from a 2023-era prompt?” is a real senior-level question. The answer above — scripted CoT, ReAct templates, aggressive nagging, prefill — signals you’ve kept current.

4.Managing prompts like code

In a production harness the prompt is a deployed artifact, and it gets the full software treatment:

a

Modular assembly

Nobody maintains one giant prompt. Claude Code assembles its system prompt from ~20 components: always-on core rules plus conditional sections (plan mode, skills, MCP) included only when relevant. Rarely-needed guidance moves to on-demand files, not the always-on prompt.

b

Versioning & registries

Prompts live in git and get commits, branches, rollbacks, and environments (dev/staging/prod). Teams use registries like Langfuse, PromptLayer, or Braintrust when non-engineers need to edit or deploys must decouple from releases.

c

Eval-gated changes

The defining 2026 practice: a prompt change must beat the current version on a test set (real traffic + edge cases, scored by checks and LLM judges) before it ships. Changing a prompt without running evals is deploying untested code.

d

A/B testing in production

Route a traffic percentage to the new prompt version, watch task success, latency, and cost per version, keep a rollback path. Prompts get canary deploys, same as services.

Anthropic’s own precondition is worth quoting: don’t prompt-engineer at all until you have success criteria and evals. Without a way to measure, you’re wordsmithing, not engineering. (Full story in Topic 7: Evals.)

5.Caching: where prompt layout becomes money

This is why the diagram in section 1 is ordered the way it is. Providers cache the prefix of your prompt — everything up to the first byte that changed:

The numbers (Anthropic): writing the cache costs 1.25× normal input price; reading it costs 0.1× — 90% off. Cache entries live ~5 minutes (1 hour costs 2× to write). OpenAI’s caching is automatic on prompts ≥1,024 tokens with a similar ~90% discount. Break-even after a single reuse.

Consequences that surprise people:

$

Long prompts are fine — if static

A 15k-token cached tool/system prefix costs less per lap than a 2k uncached one. The unit of cost is uncached tokens per request, not prompt length.

!

One timestamp can 10× your bill

Any per-request value placed early in the prompt (a date, a session ID) invalidates everything after it, every single request. The classic production caching bug.

⇅

The invalidation hierarchy

Tools → system → messages. Change a tool definition and the entire cache is cold; change the system prompt and message history caching survives above it. Harness designers order content by how rarely it changes.

Claude Code exploits this at fleet scale: every user shares one cached copy of the identical versioned system prompt. Cache hit rate is tracked as a first-class production metric.

6.Try it — upgrade a real prompt

Exercise · ~15 minutes, in Claude Code

Take the resume-rule-checker subagent from Topic 1 (or any prompt you use often) and apply three techniques from this page:

  1. Add the why: next to each rule in its system prompt, add the motivation (“resume space is limited, so…”). Rerun it on a bullet and compare strictness.
  2. Add one few-shot example: a sample bullet with the exact pass/fail output you want, wrapped in <example> tags. Watch the output format lock in.
  3. Add a self-check ending: “Before answering, re-verify each verdict against the rule text.”
  4. Meta-prompt the result: paste your improved prompt into Claude and ask “critique and rewrite this prompt for a reasoning model — remove anything obsolete.” Diff its version against yours. What did it cut?
