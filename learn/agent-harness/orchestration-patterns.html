Orchestration Patterns

One loop or many? The pattern catalog, the great multi-agent debate and how it settled, and the production craft of making several agents produce one coherent answer.
~11 min read · 1 exercise at the end
The one idea on this page Orchestration is a spectrum of autonomy: at one end, workflows — code decides the steps, models fill them in; at the other, agents — the model decides its own path. The craft is picking the least autonomy that solves the task. The 2026 field lesson is humbling: about half of the teams that started multi-agent rewrote back to a single agent with better tools. Complexity must be earned.

1.The four shapes worth knowing

The catalog has a dozen named patterns, but they compose from four basic shapes:

Everything else — routing, hierarchies, swarms — is a composition or variation of these.
→

Pipeline (prompt chaining)

Fixed, ordered steps with programmatic checks between them. Use when the decomposition is knowable in advance and you want inspectable intermediates. This is a workflow — code owns the graph.

⊞

Fan-out: sectioning and voting

Sectioning = split independent chunks (review 12 files at once). Voting = same task N times, aggregate — buy confidence with tokens (judge panels, security review). Both die when chunks share hidden dependencies.

☍

Orchestrator–workers — the production winner

A lead agent dynamically decomposes, delegates to isolated subagents, and synthesizes their compressed summaries. ~70% of 2026 production multi-agent systems. This is what Claude Code does when it spawns subagents for you.

↻

Generator–verifier (evaluator-optimizer)

One agent produces, another grades against explicit criteria, loop until pass. The trap: a verifier told only “check if it’s good” rubber-stamps everything. No articulable criteria, no verifier.

Also in the catalog: routing (a classifier dispatches to specialized paths — cheap model vs expensive model), hierarchies (supervisors of supervisors — almost everyone reaches for this too early; wait until 6+ workers), and swarms/handoffs (peers transfer control to each other — won for support-style conversations, lost for everything traceability matters in).

2.The great debate — and the synthesis

The field’s two most-cited essays landed within a day of each other in June 2025, arguing opposites:

Anthropic: parallelism wins

Orchestrator + parallel subagents beat single-agent Opus by 90.2% on research. Read-heavy work parallelizes: subagents act as intelligent filters, compressing many reads into short findings.

The catch: token spend explains ~80% of the performance gain. You buy quality with tokens.

Cognition: don’t build multi-agents

Actions carry implicit decisions; parallel agents with different partial views make conflicting decisions. Coding — write-heavy, tightly coupled — is the losing case.

Prescription: one linear agent plus great compaction. Share full traces or don’t split the work.

How 2026 settled it: both were right about their own domain, and production converged on a synthesis — one orchestrator owns the full conversation; subagents are ephemeral, isolated, and return compressed summaries. Cognition’s context discipline plus Anthropic’s parallelism. Rule of thumb: read-heavy and parallelizable → fan out; write-heavy or sequential → one agent. And a sequential multi-hop task gains nothing from more agents — parallelism, not agent count, is what buys the win.

3.Production craft

The details that separate a demo from a system:

1

Barriers vs pipelines

Barrier = wait for all workers, then synthesize (simple, but the slowest worker blocks everyone). Pipeline = stream results downstream as they finish (faster, harder to trace). Anthropic’s lead agent is deliberately synchronous — they named async as the upgrade they deferred for simplicity.

2

The task brief and the information bottleneck

Both from Topic 2b, and they’re the top two failure sources here: vague briefs make workers duplicate and drift; summaries lose critical details in handoffs. Mitigation: summaries carry pointers (file paths, trace IDs) back to full artifacts on a shared filesystem.

3

Stop loops mechanically, not by asking

Never ask an agent if it’s done looping. Detect convergence mechanically: loop-until-dry (re-spawn discovery until a round finds nothing new), hard turn ceilings, and token budgets as active safety features — not just cost controls.

4

Judges: capable, different, adversarial

A judge should be at least as capable as the agent it grades, ideally a different model (shared blind spots), scoring against a written rubric — and prompted adversarially (“find what’s wrong”), which catches edge cases neutral review misses.

5

Checkpoint, trace, and gate

Long-running orchestrations resume from checkpoints, never restart. Trace decisions, not just outputs — same prompt legitimately takes different paths. Put human approval gates at defined nodes (before writes, on low confidence), not everywhere.

4.The framework landscape, one line each

LangGraph

Explicit graph of states with checkpointing and human-interrupt nodes; the enterprise production leader. Guidance: supervisor first, swarm only if it bottlenecks.

OpenAI Agents SDK

Minimal primitives: agents, tools, handoffs, guardrails. Two blessed patterns: manager (agents-as-tools) vs handoffs. Docs say start with one agent.

Claude Agent SDK

The Claude Code harness as a library: subagents with isolated context, filesystem as shared state, hooks. What this course has been secretly teaching.

Google ADK

Code-first; cleanly splits deterministic workflow agents (Sequential/Parallel/Loop) from LLM-driven transfer.

CrewAI / AutoGen

CrewAI: role-based crews, fastest prototyping. AutoGen merged into Microsoft Agent Framework; its GroupChat peer pattern is the one that lost in production.

Protocols: MCP + A2A

The stack stratified: MCP connects agents to tools; A2A connects agents to agents across org boundaries (Linux Foundation, 150+ orgs). Interop left the frameworks and moved into protocols.

Interview angle: “Would you make this multi-agent?” The senior answer defaults to no: start single-agent, escalate only for parallelizable read-heavy work or context overflow, quote the ~15× token cost, and name the synthesis (orchestrator owns context, workers return summaries). Knowing when not to orchestrate is the signal.

5.Try it — run an orchestration and price it

Exercise · ~15 minutes, in Claude Code
  1. Pick a read-heavy question about one of your repos, e.g. “across my interview-prep repo, what are all the places resume rules are defined, and do they contradict each other?”
  2. Run it once plainly (single agent). Note the wall-clock time and, right after, run /context to see tokens consumed.
  3. New session. Same question, but: “fan out 3 parallel subagents — one for rule files, one for CLAUDE.md/skills, one for page templates — each returning a compact list, then synthesize.” That’s orchestrator-workers with sectioning.
  4. Compare: quality of answer, time, tokens. You’ll usually see the fan-out win on completeness and lose on cost — the 15× tradeoff, live.
  5. Spot the shape: which parts were a barrier (waited for all three) vs a pipeline? What did each subagent’s returned summary lose?
