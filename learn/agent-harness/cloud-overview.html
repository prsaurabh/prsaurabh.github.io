Agents in the Cloud

Topics 1–9 built an agent that runs on your laptop. This final topic moves it to the cloud, where it has to survive crashes, isolate untrusted code, talk to other agents across orgs, and stay observable and affordable at scale. Seven subpages, ordered primitives → bundles → reality.
Overview · ~4 min, then 7 subpages
Why this topic exists A local agent is a process you can watch. A cloud agent is a distributed system you can’t: it outlives its machine, runs untrusted code you didn’t write, spends real money on every lap, and may be one of thousands. Everything in Topic 10 is a consequence of that shift — and the reassuring finding from the case studies is that the winners didn’t have a smarter model, they had better harness engineering. Which is the whole course.

1.Local harness vs cloud harness

Local (Topics 1–9)

  • One process, one user, seconds to minutes
  • Crash = restart the whole thing
  • Sandbox = your OS (seccomp / sandbox-exec)
  • Tools = local MCP over stdio
  • Cost = your subscription; you watch it live
  • You are the observability

Cloud (Topic 10)

  • Many agents, minutes to days, survives deploys
  • Crash = replay from a journal, skip done work
  • Sandbox = a microVM fleet as a service
  • Tools = remote MCP + A2A over the network
  • Cost = per-token, per-tenant, at ~15× fanout
  • Observability = OTel traces + online evals

You move to the cloud for three reasons: parallelism (fan out work no laptop can hold), persistence (runs that outlive a session and survive failures), and scale (serving agents to many users, or one durable agent per user). The moment any of those is true, the seven concerns below appear.

3.The whole course, in one sentence

An agent is a dumb loop around a smart model (Topic 1), and every topic since has been a knob on that loop: what fills its context (2), how you instruct it (3), the tools it acts with (4), how many loops run at once (5), what it’s allowed to touch (6), how you know it works (7), the runtime checks around it (8), the attacks against it (9), and — here — how all of that runs in the cloud (10). The model is the brain; everything you’ve learned is the body. That body is the harness, and building it well is the job.

If you’ve read the ten topics, you can now answer the senior-interview version of “how would you build this”: not “I’d use GPT/Claude,” but “here’s the loop, here’s how I’d manage context and tools, here’s the orchestration and the evals, here’s how I’d contain and secure it, and here’s how it runs durably in the cloud.” That answer is the difference this course was built to make.
