Evals

Evals are automated tests for an AI agent. This page explains, from zero, what they are, why testing an agent is harder than testing normal code, and how teams actually do it.
~13 min read · 1 exercise at the end
The one idea on this page An eval is a test you run automatically against your AI agent, the same way a unit test checks normal code. The twist: an agent gives a differently worded answer every time you run it, so a good eval never checks the exact words. It checks what actually happened. If the agent says “I processed your refund,” the eval looks in the database for the refund. Write these tests early, and run them on every change. That is the whole discipline.
Jargon decoder — six terms used on this page
Eval
Short for “evaluation.” An automated test: give the agent a task, then use some grading logic to decide pass or fail.
Trace (or transcript)
The complete recording of one agent run: everything it said, every tool it called, and every result it got back. Like a flight recorder for one attempt at a task.
Trajectory
The path the agent took to finish: which tools it used, in what order, how many steps.
Grader
The thing that decides pass or fail. It can be a piece of code, another AI model, or a human.
LLM-as-judge
Using a second AI model as the grader. You give it the agent’s output plus a written checklist (a rubric) and ask: pass or fail?
Benchmark
A public, standardized eval suite (like SWE-bench) used to compare models against each other.

1.Why testing an agent is harder than testing code

A normal unit test is simple because code is predictable: add(2, 2) returns 4 every single time, so you test for exactly 4. An AI agent is not predictable. Ask it the same question twice and you get two differently worded answers, and sometimes two different approaches. One run says a refund takes “3 to 5 days,” the next says “within a week.” Both are correct. A test that demands exact text would fail a perfectly good agent.

Agents add two more complications. First, they take many steps, not one: a small mistake at step 2 quietly snowballs into a big failure by step 8. Second, they change things along the way: they write files, update databases, call real services. So testing an agent means answering three separate questions:

1

Did the real world end up in the right state?

This is the most important check. The classic failure: the agent tells the customer “your refund is processed” but no refund exists in the database. The words claimed success; reality disagreed. So the eval checks reality: is the database row there? Does the file exist? Do the tests pass? Never grade the agent’s claim. Grade the evidence.

2

Was the path reasonable?

Even a correct result can be reached badly: 40 steps and $3 of AI usage for a task that should take 5 steps and a few cents. A trajectory check puts limits on the path: right tools used, sane number of steps, cost under budget. One warning: don’t script the exact path. Good models find valid shortcuts you didn’t think of, and a test that demands your path will wrongly fail them.

3

Did it follow the rules while doing it?

A refund can be issued correctly and still break policy, for example if the agent never checked whether the customer was eligible. Policy checks grade the run against your business rules. This is usually where an AI judge (explained in the next section) comes in, because “did it behave appropriately” is a judgment call, not a string match.

Here is one test case graded all three ways. Notice how the agent can pass two checks and still not be safe to ship:

outcome refunds table has a row for #4411 PASS (checked by code)
trajectory 6 steps, 4 tool calls, $0.11 spent PASS (under the limits)
policy refund issued without checking eligibility FAIL (AI judge + checklist)

2.The three kinds of graders

Every eval needs a grader: something that looks at the run and says pass or fail. There are exactly three options, and the rule of thumb is use code where you can, an AI judge where you must, and humans to keep both honest.

a

Code checks — for facts

Plain code that asserts facts: the database row exists, the number matches, the unit tests pass. Fast, cheap, and gives the same answer every time. The weakness is rigidity. One real benchmark marked answers wrong for writing “96.12” instead of “96.124991”, which is a bug in the test, not the agent. Fixing grading bugs like that took one model’s measured score from 42% to 95% with no change to the model at all.

b

AI judges — for judgment calls

For fuzzy questions like “was this reply polite?” or “did it follow policy?”, you use a second AI model as the grader, with a written checklist of what pass means. Four rules make judges reliable. Ask for a yes/no verdict, not a score out of 10, because “7 vs 8” means nothing consistent. Give each judge exactly one question, never a combined overall score. Let the judge answer “not sure” instead of forcing a guess. And before trusting a judge, test the judge: grade about 100 examples by hand and check that the judge agrees with your human answers. An unchecked judge is just a second opinion of unknown quality.

c

Humans — the gold standard, used sparingly

Human review is the most accurate grading and also the slowest and most expensive. Use it for two things: creating those hand-graded examples that validate your AI judges, and periodically spot-checking that the automated graders still agree with human judgment. AI judges have known blind spots: they favor whichever answer they read first (so swap the order and average), they favor longer answers, and they favor answers written by models from their own family. Treat a judge like a measuring instrument that needs recalibration, not an oracle.

3.Run every test more than once

Because the agent behaves differently on every run, a single trial tells you almost nothing. The standard practice is to run each test case several times, and there are two very different ways to score the results:

pass@k (say “pass at k”) asks: out of k tries, did it succeed at least once? That is the right bar when a human will review the output anyway, for example research or brainstorming tasks.

pass^k (say “pass to the k”) asks: did it succeed every single time? That is the bar for anything customer-facing, because your users do not get k tries. They get one.

The gap between those two numbers is brutal, and it is just multiplication. An agent that succeeds 75% of the time per run sounds decent. But succeeding three times in a row is 0.75 × 0.75 × 0.75, which is about 42%. Your “75% agent” fails a majority of customers who use it three times.

4.Where the test cases come from

An eval suite is only as good as its test cases. In priority order, the best sources are: real production failures (a user hit this, it broke), real production successes (so you notice if they break later), edge cases you write by hand, deliberately hostile inputs, and only then synthetic filler generated by a model.

Three quality rules for the cases themselves. Keep them unambiguous: a good test case is one where two experts, working separately, would reach the same pass/fail verdict. Keep them balanced: if you are testing “should the agent search the web?”, include cases where the right answer is no, or the agent learns to always search. Keep them solvable: if nothing ever passes a case, suspect the case before the agent.

Don’t wait for the perfect suite. Teams stall for months thinking they need hundreds of polished cases. In practice 20 to 50 real tasks, pulled from actual failures, is a genuinely useful starting suite. It grows from there.

5.The everyday workflow: read, diagnose, automate, repeat

Where do you even find your agent’s failure patterns? Not by imagining them. The practitioners who do this well follow a loop called error analysis, and it works like a doctor diagnosing before prescribing: examine the patient first, then treat what you actually found.

1

Read the traces

Collect around 100 recorded runs and actually read them. For each failed run, write one plain sentence: what was the first thing that went wrong? Not the last error on screen, the first upstream mistake that caused it. This is the single most repeated advice in the field, and the most skipped. Top teams spend well over half their eval time here.

2

Group the failures, then automate the top ones

Cluster those one-sentence notes into categories: “picks the wrong tool,” “gives up too early,” “invents order numbers.” Now build an automated grader for each of the biggest categories. You are building tests for failures you have actually seen, not failures you imagined. Generic scores like “helpfulness: 7/10” measure nothing.

3

Run the suite on every change

Wire the suite into your build pipeline so it runs automatically on every prompt edit, tool change, and model upgrade, and blocks the change if cases regress. This is exactly how teams like Notion, Stripe, and Vercel ship agent changes without holding their breath.

4

Keep watching in production

After shipping, run the cheap code checks on all live traffic and the AI-judge checks on a small sample of it. Whatever fails out there becomes next week’s new test cases. The loop closes and the suite keeps getting more realistic.

One caution about public benchmarks. Leaderboard scores rank models, not your agent. Public suites get saturated fast (SWE-bench went from hard to ~95% solved within about a year), and a 2026 Berkeley study showed eight major agent benchmarks could be gamed to near-perfect scores without really solving the tasks. Use benchmarks to shortlist a model. To know whether your agent works, there is no substitute for your own suite. Popular tools for the recording-and-testing infrastructure: LangSmith, Langfuse (open source), Braintrust, and Arize Phoenix.

6.The mistakes everyone makes

Never reading transcripts

Dashboards and scores feel like progress, but until you read real runs you don’t know if your graders even work.

Vague, generic metrics

“Helpfulness” and “coherence” scores sound rigorous and measure nothing. Test for the specific failures you observed.

One combined score

A single “7.2/10” can hide that accuracy got worse while tone got better. Keep one grader per question.

Teaching to the test

Tune against the same fixed cases long enough and scores improve while the real agent doesn’t. Hold some cases back and keep adding fresh ones from live traffic.

Over-strict graders

Failing an agent for harmless differences (formatting, a valid alternate approach) means the test is the bug, not the agent.

Waiting for perfect

20 real test cases today beat 500 imaginary ones next quarter. The payoff compounds: you stop hand re-testing everything at every model release.

You’ll hear the slogan “evals are the moat.” The sober version: the models you build on will keep changing under you, but your library of real test cases, your catalog of known failure patterns, and your validated judges keep their value. That accumulated testing infrastructure is the durable asset.

7.Try it — eval your own agent

Exercise · ~20 minutes, in Claude Code

Build a real (tiny) eval suite for the resume-rule-checker you made in Topic 1:

  1. Write 6 test bullets in a file: 3 that follow all your resume rules, and 3 where you plant exactly one violation each (a dash, a “Technologies:” line, an over-length rewrite). You now have a small, balanced dataset where you already know every correct answer.
  2. Run the checker on all 6. Grade each one yes/no: did it reach the correct verdict? Also note which planted violation it caught or missed. That per-violation note is component-level grading.
  3. Run the same 6 again. Any bullet whose verdict flipped between the two runs is non-determinism, live. That’s why one run is never enough.
  4. Take the worst case, read both of its transcripts, and write one sentence naming the first thing that went wrong. That sentence is error analysis, and it tells you exactly what to fix in your prompt next.
