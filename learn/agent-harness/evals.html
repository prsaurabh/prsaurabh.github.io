1.Why agent evals are a different animal

A model eval is single-shot: prompt in, response out, grade it. An agent takes many laps (Topic 1) that modify state — mistakes compound, and the same task legitimately takes different paths on different runs. Three consequences:

1

Grade the end state, not the words

The canonical failure: the agent says “your refund is processed” and no refund exists in the database. Check the environment — the DB row, the file, the passing tests — not the claim. And grade what was produced, not the path taken: strong models find valid paths you didn’t anticipate, so over-specified trajectory checks create false failures.

2

Run every task multiple times: pass@k vs pass^k

Non-determinism makes one trial meaningless. pass@k = at least one of k runs succeeds (fine for research tasks). pass^k = all k succeed (what customer-facing agents need). The gap is brutal: 75% per-run is only ~42% pass^3.

3

Three altitudes of grading

End-to-end (did it succeed?), trajectory (was the path sound — right tools, sane turn count, token budget?), and component (which tool or subagent broke?). Production suites run all three; the fanciest benchmark, tau-bench, adds a fourth: did it succeed while following policy?

2.The eval stack: datasets and graders

Datasets, in priority order: sampled real production traces, user-reported failures, hand-written edge cases, adversarial cases, synthetic fill. Balance positives and negatives (a “should it search the web?” eval needs cases where the answer is no). The quality bar for a task: two domain experts would independently reach the same pass/fail verdict — and if nothing ever passes, suspect the task before the agent.

Graders, three tiers — the rule is deterministic where possible, LLM judges where necessary, humans judiciously:

a

Deterministic checks

String match, unit tests, state assertions, tool-call verification. Fast, cheap, reproducible — but brittle: one benchmark failed answers for writing “96.12” instead of “96.124991”, and fixing grading bugs alone took one model from 42% to 95%.

b

LLM-as-judge — designed, not just prompted

Binary pass/fail beats 1–5 scales (clearer thinking, consistent labels). One isolated judge per dimension, never a composite score. Give the judge an escape hatch (“return Unknown”). And validate the judge itself against ~100 human-labeled examples — an unvalidated judge is just vibes with extra steps.

c

Know the judge’s biases

Position bias (swap the order and average), verbosity bias (longer ≠ better), self-preference (use a different model family as judge), and calibration drift (recalibrate against humans on a cadence). Frontier judges exceed 50% error on bias stress tests — treat judges as instruments that need calibration, not oracles.

3.The lifecycle: CI → production → back again

1

CI for agents

The eval suite runs on every PR — prompt change, tool change, model upgrade — and posts per-case regressions before merge. This is how Notion, Stripe, and Vercel ship agent changes; Notion credits the loop with a 10× jump in issue-triage throughput.

2

Online evals: monitoring is a continuous eval

Cheap deterministic checks on 100% of traffic; sample 5–10% for LLM-judge scoring; A/B test big changes. Failures found here become next week’s eval cases — the loop closes.

3

Public benchmarks rank models, not your agent

SWE-bench Verified saturated (~95%) within a year; a 2026 Berkeley study exploited eight major agent benchmarks to near-perfect scores without solving the tasks. And simulated-user evals swing up to 9 points depending on which model plays the user. Use benchmarks for model choice; build your own suite for your harness.

Tracing comes first. 89% of teams have observability; only 52% do evals — the gap is the opportunity. You cannot build eval datasets without traces. The practitioner workflow: collect ~100 traces → annotate each with its first upstream failure → cluster into a failure taxonomy → build judges for the failures you actually observed, not the ones you imagined. Tools: LangSmith, Langfuse (open source), Braintrust (strongest CI loop), Arize Phoenix — all converging on OpenTelemetry’s still-maturing GenAI standard.

4.The mistakes everyone makes

Skipping transcript reading

The most-repeated advice in the field: you won’t know your graders work until you read many trials’ transcripts and grades.

Generic metrics

“Helpfulness” and “coherence” scores measure nothing. Build judges from your observed failure taxonomy.

One composite score

A single 7.2/10 hides that citations regressed while tone improved. Separate judges per dimension.

Overfitting to the suite

Optimizing against the same cases improves scores, not reality. Hold out cases, refresh from fresh traffic, watch for grader loopholes.

Rigid graders

Penalizing valid variation (formatting, equivalent paths) makes the eval the bug.

Waiting for the perfect suite

20–50 real cases beat 0 perfect ones. Costs are visible upfront; the benefit — not manually re-testing at every model release — compounds forever.

On “evals are the moat”: the sober version is that evals are compounding infrastructure — the durable asset is your failure taxonomy, labeled data, and calibrated judges, which persist while the underlying models churn.

5.Try it — eval your own agent

Exercise · ~20 minutes, in Claude Code

Build a real (tiny) eval for the resume-rule-checker from Topic 1:

  1. Write 6 test bullets in a file: 3 that pass all rules, 3 with one planted violation each (a dash, a “Technologies:” line, an over-length rewrite). That’s your dataset — balanced, with known answers.
  2. Run the checker on all 6. Grade binary per bullet: did it reach the correct verdict? Note it caught/missed which planted violation — that’s component-level grading.
  3. Run the same 6 again. Any bullet that flipped verdicts between runs is your pass^k lesson, live.
  4. Read the two transcripts for the worst case. Write one sentence: what was the first upstream failure? Congratulations — you’ve done error analysis, and that sentence is your next prompt fix (Topic 3’s exercise, now with a measurement behind it).
