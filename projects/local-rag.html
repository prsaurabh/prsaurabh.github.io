LocalRAG

Privacy-Focused Retrieval-Augmented Generation

Overview

Dual-mode RAG application designed to demonstrate traditional RAG fundamentals. Public mode leverages AWS Bedrock for cloud-based inference, while Private mode runs fully local with Ollama, ensuring no data ever leaves your machine. The system supports PDF, DOCX, TXT, and image files with OCR-based text extraction, giving users flexibility across document formats while maintaining control over data privacy.

Architecture

Documents (PDF, DOCX, TXT, Images)
        ↓
Text Extraction (PyMuPDF + OCR)
        ↓
Chunking → Embedding → ChromaDB
        ↓
Query → Vector Search → Top-K Chunks
        ↓
LLM (Bedrock or Ollama) → Answer

Dual Mode Design

Public Mode uses AWS Bedrock Claude for generation and Bedrock Vision for OCR. Documents are sent to AWS for processing, which provides higher accuracy and faster inference but requires cloud connectivity and an AWS account.

Private Mode runs entirely on your local machine using Ollama for LLM inference and Tesseract for OCR. No data leaves the device. This mode trades some speed and accuracy for complete data privacy -- ideal for sensitive or confidential documents.

The privacy trade-off is straightforward: Public mode gives you access to frontier models and vision capabilities at the cost of sending data to AWS. Private mode keeps everything local at the cost of requiring local compute resources and accepting the limitations of smaller open-source models.

Key Features

  • Multi-format document support -- PDF, DOCX, TXT, and images (PNG, JPG) are all handled through a unified text extraction pipeline.
  • PyMuPDF for PDF processing -- Replaced the original Poppler dependency with PyMuPDF (fitz), eliminating a complex native library dependency and simplifying installation across platforms.
  • Local embeddings with nomic-embed-text -- Both modes use the same local embedding model, keeping vector representations consistent regardless of the inference mode selected.
  • Persistent ChromaDB storage -- Document embeddings are stored on disk and survive application restarts, avoiding re-ingestion of previously processed documents.
