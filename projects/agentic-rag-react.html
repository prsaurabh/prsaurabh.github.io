Agentic RAG - ReAct Agent

Autonomous Tool Selection with LlamaIndex

Overview

A LlamaIndex ReAct agent that autonomously reasons about which tools to use for any given query. Unlike a fixed pipeline, the LLM itself drives the decision loop -- Reason, Act, Observe, Repeat -- iterating through available tools until it has gathered enough context to produce a final answer. The agent decides when to search the vector store, grep through files, read full documents, or query the web, all without hard-coded routing logic.

Architecture

User Query
    ↓
ReAct Agent (Bedrock Claude)
    ↓
[Reason] What do I need?
    ↓
[Act] Select Tool:
├── vector_search → ChromaDB
├── file_search    → Keyword grep
├── file_read     → Full file content
└── web_search    → Tavily API
    ↓
[Observe] Process result
    ↓
Loop or Answer

Key Design Decisions

LlamaIndex v0.14 AgentWorkflow -- Built on the latest AgentWorkflow API rather than the deprecated ReActAgent class. This provides a cleaner interface for defining tools and handling the agent loop with proper async support.

BedrockConverse for async streaming -- Uses the BedrockConverse LLM adapter which supports async streaming natively, enabling real-time token-by-token output in the Streamlit frontend as the agent reasons through its steps.

Local embeddings -- Despite using Bedrock for inference, embeddings are generated locally with nomic-embed-text via Ollama. This keeps the embedding layer consistent across projects and avoids per-query embedding costs.

Priority-based system prompt -- The agent's system prompt establishes a clear tool priority order: vector search first, then file search, then file read, with web search as a fallback. This guides the agent toward the most efficient retrieval path without hard-coding the logic.

MCP Server

The project includes a Model Context Protocol (MCP) server that exposes ChromaDB as a set of tools for external clients. Two tools are provided: search_documents for semantic vector search and list_documents for browsing the document collection. The server supports both stdio and SSE transport, making it compatible with Claude Desktop and other MCP-aware clients that need access to the project's knowledge base.
