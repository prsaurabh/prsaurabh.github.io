Overview

What this platform does and why it exists

Anywhere Real Estate operates multiple business lines — title & settlement, franchising, brokerage, relocation, and mortgage services. Each group maintains distinct tech stacks. As AI adoption accelerated, teams were independently integrating with vendors, creating fragmented, ungoverned implementations with no shared infrastructure, no cost visibility, and no consistent safety.

The Enterprise AI Foundational Service replaced that fragmentation with a unified, governed platform. Every LLM call across the enterprise flows through this platform — from simple completions to multi-agent workflows. It offers capabilities via REST API, SDK (npm, NuGet, PyPI), MCP servers, and a self-service Engineering Portal.

The core design principle is configuration over code: teams change AI behavior — models, prompts, safety settings, RAG parameters, tool connections — through the portal without redeploying their applications. This is what makes it a platform, not just an API wrapper.

Agent Framework

The primary execution model — self-contained AI units

An Agent bundles an LLM, system instructions, tools, safety settings, and temperature into one deployable configuration. Consumers invoke with a single agentId — the LLM autonomously decides which tools to call based on the query. The platform recommends Agents over older pipeline-based execution for all new development.

Response Format & Typed Contracts JSON Schema

The Response Format feature converts JSON Schema definitions to system prompt instructions at runtime. This creates typed data contracts for agent output — enabling reliable agent-to-agent chaining where one agent's structured output feeds directly into the next agent's input.

Agent-to-Agent Composition Recursive

Agents are registered as tools in the Tool Catalog alongside RAG, MCP, and OpenAPI tools. Any agent can invoke any other agent — recursive composition to arbitrary depth. The LLM drives routing based on tool metadata. This pattern anticipates Google's A2A protocol (now a Linux Foundation project with 150+ partners).

Streaming & Real-Time SSE

SSE-based streaming with 5 event types: chunk, tool_request, tool_response, complete, error. The full tool-use lifecycle is visible in real time — consumers can show users what the agent is doing as it reasons through tool calls.

04

Universal Tool Catalog

The most consequential architectural decision

A single registry holding 4 tool types — discoverable and attachable through one configuration surface. Agents consume tools regardless of type. New tool types can be added without changing how agents work. Registration in the portal is the governance gate.

Tool TypeWhat It DoesUse Case
RAGManaged retrieval over uploaded documentsLLM decides when to search based on query
MCP ServerCustom tool logic via Model Context ProtocolBespoke integrations, data access, actions
OpenAPIUpload spec, LLM calls endpoints autonomouslyStandard REST APIs, no server code needed
AgentAnother agent registered as a toolRecursive agent composition to arbitrary depth
Tool Versioning Semver

Semantic versioning for MCP Servers and OpenAPI Tools. New versions clone all settings; existing agents stay pinned to their version. Blue-green deployment pattern for AI tooling. Non-breaking contract enforcement: tools referenced by agents can't be removed or renamed, but descriptions and schemas can evolve.

Why this matters: Before the Tool Catalog, each integration type had its own configuration surface, discovery mechanism, and attachment pattern. Unifying them means an agent doesn't care whether a tool is a RAG index, an MCP server, a REST API, or another agent — it just calls it.

Platform Core

Model routing, prompt management, memory, and cost optimization

Model Access & Routing
Dynamic Model Routing Multi-Cloud

Intelligently routes requests to optimal models based on task requirements, cost, and performance metrics. The unified API normalizes differences between AWS Bedrock, Azure OpenAI, and open-source models — switching providers requires only changing a model identifier.

  • Provider abstraction — single interface across Claude, Llama, Mistral, Titan, Cohere
  • High-availability failover — automatic rerouting when a provider degrades
  • A/B testing — controlled rollout of new models with benchmarking against existing ones
  • Local model fallbacks — reduces cloud dependency for suitable workloads
Prompt Management & Versioning Ceiling-Based Semver

Versioned prompt settings with ceiling-based semver resolution: clients request a max version, the platform resolves the latest at or below. Prompt engineers publish without client redeployment. Breaking changes require explicit version bump. Instant rollback.

  • Template variables — {data} for runtime JSON, {documents} for RAG content, {{mustache}} for platform-injected context (date/time)
  • Thread-scoped injection — system prompt applied only on thread creation, not repeated every turn
  • Per-LLM settings — different system instructions optimized for each model family
  • Privacy mode — sensitive workflows can opt out of logging prompt content
  • Self-service administration — Admin, Editor, Viewer roles with resource-level permissions
Memory Management Context-Aware

Intelligently compresses and reuses chat history within context windows, reducing unnecessary token usage. Retains user interaction history for long-term AI memory across multi-turn conversations.

Dynamic KV Cache Tuning Agent Module

The platform allows dynamic LLM selection, and an agent module optimizes context utilization per model:

  • Identifies the tokenizer used by the selected model (tiktoken, sentencepiece, etc.)
  • Fetches the model's token limit (8K, 32K, 128K)
  • Adjusts chunk size, overlap, and strategy accordingly
  • Tunes retrieval parameters (top-k, score threshold) to fit tighter or looser contexts
  • Optionally switches ensemble methods (hybrid, rerank, multi-query) based on model cost vs quality
Cost Optimization
Cost Controls & Efficiency Caching + Limiting
  • Semantic caching — caches frequently queried AI responses to reduce API calls and latency
  • Intelligent rate limiting — dynamically scales down non-critical requests to prevent excessive token burn
  • Batch processing & parallel execution — aggregates requests into concurrent processing pipelines
  • Token-by-token streaming — optimized for real-time interactions with minimal delay
  • Experiments Playground — side-by-side comparison of up to 4 models with independent config, actual token usage and cost after execution, and a pre-execution pricing calculator

Workflow Orchestration

Multi-agent DAGs with conditional branching

Multi-step workflows defined as directed acyclic graphs with 4 step types:

Step TypeWhat It Does
TriggerManual API invocation or scheduled (cron). Webhook support planned.
AgentInvoke any registered agent from the catalog.
FilterConditional branching via JSON path evaluation — equals, not-equals, contains, greater-than. Enables fan-out with predicate guards.
NotificationEmail delivery at any point in the workflow.

Async execution with full step-level input/output traceability. Each step's output feeds into the next via typed schema contracts, enabling complex multi-agent pipelines where different agents handle different stages of a business process.

MCP & Agent Marketplace

Reusable agents as services with dynamic discovery

Inspired by Agent.ai and n8n, the Agent Marketplace makes agents available as services, SDKs, and MCP servers. The vision is to scale and add a visual builder and orchestrator once enough skills are available on the platform — so even product managers or non-developers can build agents.

MCP Implementation

The platform's agents and systems are built using different technologies and frameworks — C# (Semantic Kernel, AutoGen), Python (LangChain, LangGraph), and Java (AWS Bedrock agents). MCP serves as the interoperable protocol integrating these systems while giving data owners control through their MCP servers.

MCP Registry: Agents dynamically discover available tools and resources across the organization. Data owners expose resources via their own MCP servers, maintaining control over what's accessible.
Central Authorization Okta + OAuth2

The JSON-RPC message format with server-side events connection that MCP uses required building a custom Okta adapter to implement central authorization from local MCP to cloud-hosted clients.

Hierarchical Chaining Multi-Agent

MCP agent chaining for data and tools aggregation with adapters from lower systems. Enables complex, multi-step AI reasoning pipelines by integrating external APIs and automation tools.

Agent Types Marketplace
  • Document Processing Agent — extraction, parsing, structure recognition
  • Knowledge Base Agent — RAG-powered Q&A
  • Multi-Modal Agent — vision, image captioning, OCR
  • Orchestration Engine — pipelines and workflows
  • Browsing Agent — evolved from custom Playwright to Browser-use, Anthropic Computer Use, and Amazon Nova Act
Runtime Infrastructure Cross-Stack

Rate limiting, retry configuration, logging, and guardrails are implemented at both client and server level across runtimes. Also evaluating emerging protocols like IBM ACP and Google A2A for future interoperability.

Safety & Compliance

An architectural invariant, not a developer option

Non-Optional by Construction: Every API request MUST reference a Safety Setting. There is no unguarded path through the platform. This is enforced at the architectural level — not by developer discipline. Dual-provider defense-in-depth: AWS Guardrails (policy-based) + Azure AI Content Safety (ML-based). Bidirectional filtering on both inputs and outputs.

Platform Evolution

From one team's request to enterprise-wide standard

Phase 1
Standardize Model Access
Started with a focused ask — standardize OpenAI access for one team. Built the initial gateway with provider abstraction and unified API.
Phase 2
Core Platform Capabilities
As AI adoption accelerated, teams wanted different models, different providers, cost controls, guardrails, RAG patterns. Recognized this wasn't about one integration but organizational AI governance. Built prompt management, memory, caching, and the observability backbone.
Phase 3
RAG & Agent Services
Decomposed RAG into an agent-based pipeline offered as a service. Built the agent marketplace, published SDKs, and launched the pipeline builder UI for self-service RAG.
Phase 4
MCP & Cross-Team Adoption
Implemented MCP as the interoperability protocol across C#, Python, and Java runtimes. Built the central MCP registry with Okta authorization. Platform became the standard for 8+ product teams, enabling 23 active applications.
Current
Orchestration & Builder
Scaling toward a visual builder and orchestrator. Exploring multi-agent orchestration at scale, browsing agents (Anthropic Computer Use, Amazon Nova Act), and additional agent communication protocols (IBM ACP, Google A2A).
