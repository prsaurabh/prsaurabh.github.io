Agentic RAG - Orchestrated

Developer-Controlled Pipeline with LangGraph

Overview

A LangGraph state machine where the developer explicitly defines the execution flow. Unlike the ReAct approach where the LLM decides what to do next, here the graph controls routing between nodes. Each node calls the LLM for a specific, scoped task -- classify, grade, rewrite, generate -- but the transitions between nodes are deterministic and developer-defined. The system includes self-correction through document grading and query rewriting, ensuring answer quality without unbounded agent loops.

Architecture

START
  ↓
Router (LLM classifies: vectorstore or web_search)
  ↓
Retriever (ChromaDB or Tavily)
  ↓
Grader (LLM: are docs relevant?)
  ├── Yes → Generator (LLM synthesizes answer)
  │           ↓
  │         Hallucination Check (LLM: is answer faithful?)
  │           ├── Yes → END
  │           └── No → Generator (retry, max 1)
  └── No → Rewriter (LLM rewrites query)
              ↓
            Retriever (loop, max 2 retries)

Self-Correction Loops

The graph implements two distinct self-correction mechanisms, each with bounded retries to prevent infinite loops:

Grader-Rewriter Loop (max 2 retries) -- After retrieval, the Grader node asks the LLM whether the retrieved documents are relevant to the query. If the documents are judged irrelevant, the Rewriter node rewrites the query to improve retrieval, and the system loops back to the Retriever. This can happen at most twice. After two failed attempts, the graph forces generation with whatever documents are available rather than looping indefinitely.

Hallucination-Regeneration Loop (max 1 retry) -- After the Generator produces an answer, the Hallucination Check node asks the LLM whether the answer is faithful to the retrieved documents. If the answer is flagged as hallucinated, the Generator runs once more with the same context. After one retry, the graph accepts the output and terminates. This prevents the system from endlessly regenerating while still catching obvious hallucinations.

6 Nodes, 5 LLM Calls

The graph consists of six nodes, five of which invoke the LLM for distinct tasks:

  • Router -- LLM classifies the query as requiring vector store retrieval or web search.
  • Retriever -- No LLM call. Fetches documents from ChromaDB or Tavily based on the router's decision.
  • Grader -- LLM evaluates whether retrieved documents are relevant to the original query.
  • Rewriter -- LLM rewrites the query to improve retrieval results on the next attempt.
  • Generator -- LLM synthesizes a final answer from the retrieved context.
  • Hallucination Check -- LLM verifies the generated answer is grounded in the retrieved documents.
