Evals for the Developer Intelligence Platform

The platform is not one agent, it is a fleet, coding, testing, change-risk, troubleshooting, orchestration, and secure execution, each running distributed across pipelines and tools. Every one of them is non-deterministic, so the eval is the control system that decides whether it can be trusted to act. This page is the hub: the shared eval discipline, then how you eval each use case. Coding and PR agents get their own deep-dive; general theory is in Evals fundamentals.
The one idea You do not eval "the platform" with one number. Each use case leaves a different end state in a real system, a merged diff, a caught fault, a routed incident, a contained blast radius, and you grade that, per use case, against labeled ground truth, gated into CI. As the agents get smarter and vendors ship more out of the box, the durable engineering asset is not the agents, it is the per-use-case failure taxonomies, labeled traces, and calibrated judges that keep the fleet honest while the models churn.

1.The shared discipline (applies to every use case)

Grade the end state, not the words

Check the environment, the merged code, the DB row, the passing tests, not the agent's claim. Strong models find valid paths you did not anticipate, so grade what was produced, not an over-specified trajectory.

Run it k times: pass^k, not one lucky trial

Non-determinism makes a single run meaningless. Customer-facing steps need all-k-succeed (pass^k), and 75% per-run is only ~42% for three in a row. This is why an agent step is not a normal CI step.

Eval-driven development, gated in CI

Write the eval first, build until it passes, and gate every prompt, tool, and model change on the suite. Offline labeled sets for regression on every PR; online deterministic checks on all traffic plus judge scoring on a 5–10% sample. Online failures become next week's offline cases.

Tracing comes first

You cannot build datasets without traces. Per touchpoint: collect ~100 traces, annotate each with its first upstream failure, cluster into a taxonomy, build judges for the failures you actually saw. 20–50 real cases beat a perfect suite that never ships.

2.How you eval each use case

⌨️

Code generation & review

End state: diff compiles, tests pass, review caught real defects without a false-positive flood.
Metric: precision/recall on a labeled PR set, plus trajectory, patch locality and collateral risk (did the fix quietly break something else).
2026: SWE-bench Verified saturated (~95%), so public benchmarks pick your model, not your agent. Build your own suite. Deep-dive →
🧪

Intelligent testing (test generation)

End state: generated tests fail on injected faults and pass on correct code; the risk model picked the tests that matter for the change.
Metric: mutation score, not coverage. High coverage does not imply fault detection; mutation score is the reliable signal and is becoming a default in 2026.
Trap: AI-written tests have high assertion density but agents favor value-revealing print statements over real assertions, and introduce test smells like Assertion Roulette. Grade fault-catching, not lines touched.
🔎

Change & blast-radius analysis

End state: the risk score matches what actually broke, low-risk changes auto-merged were genuinely safe, risky ones were flagged.
Metric: precision/recall against historical incidents, plus pre-deployment validation rate (share of changes safely simulated first). Blast radius = access scope × operating velocity × detection window.
2026: blast radius is the underrated deployment metric, control all three factors, not just the model.
🚨

Production troubleshooting (AIOps)

End state: the linked root cause matches the real one and it routed to the owning team.
Metric: three axes, alert-noise reduction (70–95% achievable), root-cause accuracy scored on on-call agreement, and MTTR reduction (20–40%, up to ~60%).
Trap: never trust the first root-cause suggestion blindly, grade it against resolved incidents with a known cause, not whether the summary reads well.
🔗

Workflow orchestration (multi-agent)

End state: the chain completed with correct hand-offs and the right human-in-the-loop gates fired.
Metric: grade the whole trajectory, plus two cheap health signals, token cost per workflow (a 10× spike flags retry loops or context flooding) and agent-to-human escalation rate.
Trap: a sudden drop in escalation rate is not "the agent got better", it usually means a guardrail was bypassed. Watch it as a safety signal.
🛡️

Secure execution of AI code

End state: AI-generated code ran isolated and could not touch anything real until it passed through a PR.
Metric: isolation incidents (target zero), least-privilege coverage, and a containment-confidence measure that boundaries hold during failure.
2026: in a connected fleet a single compromised agent is a vector into every agent that trusts it, so this eval gates all the others.

3.Build vs buy for the eval layer

The platform's own pattern applies to evals too, buy the engine, build the part specific to you:

Buy the engine

Eval runner, trace store, judge scaffolding, CI integration, commodity now (Braintrust, DeepEval, LangSmith, Langfuse, Arize Phoenix). Building your own runner is undifferentiated work a funded vendor does better.

Build the moat

Per-use-case failure taxonomies, labeled traces from your pipelines, golden datasets, and calibrated judges. None of that transfers from a vendor, it is what keeps your fleet on track, and it compounds while the models churn.

The frontier moves. As agents sharpen and vendors ship more eval features out of the box, the build/buy line slides toward buy. Build behind an abstraction so you can retire your own runner when a product catches up, but keep owning the taxonomy and the labeled data.

4.Why this is the skill that's appreciating

The counter-intuitive shift: the smarter the fleet gets, the more the senior human's value concentrates in evaluation. When agents can do most of the work across all six use cases, the scarce skill is defining what "good" means precisely enough to grade it, per use case, and gating the fleet on it. The teams shipping agents successfully in 2026 are not the ones with the best models, they are the ones with the best evaluation infrastructure.
