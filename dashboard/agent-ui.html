Extending Agents to UI

When an agent's answer should be a form, a dashboard, or an approval panel, not a wall of text. As of mid-2026 this settled into a layered protocol stack rather than one winner. The central, extensible choice for a platform already built on MCP is MCP Apps, the official MCP UI extension, and this page leads with it, then covers the rest of the stack and when to use each.
The one idea Agents outgrew plain text. The question is who draws the UI and where it runs. Four protocols answer different halves of that: MCP Apps lets a tool return interactive UI into a host (chat or IDE); A2UI lets an agent draw a native UI itself; AG-UI streams an agent's state into a frontend you own; A2A is agent-to-agent. If you are already MCP-central, MCP Apps extends that investment instead of adding a parallel UI stack.

1.MCP Apps the primary choice

MCP Apps is the official UI extension to the Model Context Protocol (shipped 2026-01-26). It consolidated two earlier efforts, the community MCP-UI project and OpenAI's Apps SDK, into one open standard, so a tool can return an interactive UI component instead of plain text. It is the natural fit here because the platform already speaks MCP everywhere.

Three components

MCP Serverdeclares the UI (HTML) + tools, returns a ui:// resource
→
HostClaude / VS Code / ChatGPT: embeds the View, proxies calls
→
Viewsandboxed iframe UI: renders, calls tools back

The tool returns a ui:// resource; the host renders it in a sandboxed iframe; the View and host talk over JSON-RPC 2.0 via postMessage. Every View-to-host message is an auditable MCP JSON-RPC call, and the host stays in control of everything the View can do.

Why it's safe enough for enterprise

Sandbox isolation

The View runs in a restricted iframe, no access to the parent DOM, cookies, or localStorage, no navigating the parent, no scripts in the host context. It can only speak through postMessage.

Restrictive-by-default networking

Servers declare the domains their UI needs via CSP metadata, and hosts enforce it. Declare nothing and no external connection is allowed, which shuts down data exfiltration to undeclared servers.

Pre-declared, reviewable templates

Tools declare their UI templates ahead of time, so a host can prefetch, cache, and security-review them before anything runs, rather than rendering arbitrary code on the fly.

Graceful degradation

Hosts advertise UI support on connect; servers check before registering UI tools. On a host without MCP Apps, the same tool just returns text. Nothing breaks.

What teams build with it

Approval & review panels

An infra change rendered as diagrams, security-group tables, IAM diffs, and cost estimates, so an approver sees what changes and what depends on what, inline. A natural fit for gating agent actions in a dev platform.

Dashboards & data exploration

Interactive charts and sortable tables the user filters and drills into without leaving the conversation or exporting anything.

Forms & multi-step workflows

Structured input and guided flows instead of the agent parsing free text, which cuts a whole class of misunderstanding.

Where it's shipping. Hosts: Claude, ChatGPT, VS Code, Goose. In VS 2026 the Azure MCP Server became first-party, managing Azure from natural-language prompts in the IDE. Launch partners include Figma, Slack, Canva, Asana, Box, monday.com, Hex, Salesforce. A 2026-07-28 MCP spec release candidate continues to fold the extension into the core standard.

2.The rest of the stack, and when to use each

MCP Apps covers "a tool returns UI into a host." The other three protocols answer different halves of the agent-UI problem, and they compose rather than compete:

A2UI

Agent-to-UI · Google + CopilotKit · Apache-2.0

Use when the agent should draw the interface itself. A streaming protocol where the agent emits a blueprint of native components, not arbitrary code, so the UI renders natively across web, mobile, and desktop and inherits the host app's styling and accessibility. Newest of the four, and explicitly designed to combine with MCP Apps.

AG-UI

Agent-User Interaction · CopilotKit

Use when you own the frontend and need a live, stateful session. An event-based transport that streams an agent's execution state, messages, tool calls, state patches, lifecycle signals, into a frontend application you control, as it happens.

A2A

Agent-to-Agent · Linux Foundation

Use for agent-to-agent coordination, not UI at all. Included because it is the other half of the stack: discovery, task lifecycle, and messaging between agents across frameworks.

The mental model. A2A (agent ↔ agent) → MCP (agent ↔ tools) → AG-UI (agent ↔ your frontend runtime) → A2UI (agent ↔ a UI it draws). MCP Apps is the "tool returns UI inside a host" path within MCP. There is no single winner, you pick the layer by who draws the UI and where it runs. For an MCP-central platform, MCP Apps is the extensible default, with A2UI as the declarative complement.

3.Why MCP Apps was the right central bet here

The platform already exposes everything, repos, CI/CD, internal systems, over MCP. Adding UI through MCP Apps means one protocol, one security model (the same sandbox and capability negotiation), and one integration surface, instead of a second UI stack to build, govern, and maintain. It degrades gracefully on hosts that lack it, it is standardized rather than proprietary, and it rides an ecosystem the whole industry is converging on. That is the "scalable, central, extensible" choice, buy into the standard, keep A2UI in view for the cases MCP Apps does not cover.
